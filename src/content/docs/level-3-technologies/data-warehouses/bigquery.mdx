## BigQuery: Google's Serverless Data Warehouse

**BigQuery is Google Cloud's fully managed, serverless data warehouse — its interview questions center on how separating storage (Capacitor) from compute (Dremel) shapes partitioning, clustering, slot-based performance, and cost control.**

## Key Concepts

BigQuery's defining trait is a serverless split between storage (**Capacitor**, a columnar format) and compute (**Dremel**, its distributed query engine), so users never manage clusters — they just query, and BigQuery scales both layers independently behind the scenes. Performance and cost both hinge on scanning less data: **partitioning** (typically by ingestion time or a date/timestamp column) and **clustering** (physical co-location by column values within a partition) let the query engine skip irrelevant blocks entirely. Compute is measured in **slots** — units of query processing capacity, billed either on-demand (pay per byte scanned) or via **flat-rate reservations** for predictable cost at scale. Data can land via batch loads, the **streaming API** (near-real-time, buffered before permanent storage), or federated queries against external sources. BigQuery natively supports **nested and repeated fields** (structs and arrays), avoiding the joins a normalized relational schema would require. Governance rests on **IAM** roles, audit logs, and metadata tools like **INFORMATION_SCHEMA** and the Data Catalog, while **BigQuery ML** lets you train and evaluate models using only SQL.

<QuestionCard id="bq-fund-04" number={4} question="What is the difference between BigQuery and traditional relational databases?" difficulty="easy" topic="Fundamentals">

<QuestionCard id="bq-fund-05" number={5} question="Explain the concept of federated queries in BigQuery." difficulty="medium" topic="Federated Queries">

<QuestionCard id="bq-fund-06" number={6} question="What is the difference between a table and a view in BigQuery?" difficulty="easy" topic="Views">

<QuestionCard id="bq-fund-07" number={7} question="Explain the concept of nested and repeated fields in BigQuery." difficulty="medium" topic="Fundamentals">

</QuestionCard>

<QuestionCard id="bq-fund-08" number={8} question="What are the limitations or constraints of using BigQuery?" difficulty="easy" topic="Fundamentals">
Some limitations of using BigQuery include:
- Query costs: Large or complex queries can result in higher costs.
- DML operations: BigQuery does not support traditional update and delete operations on tables.
- Data consistency: BigQuery is designed for analytical workloads and does not provide strong transactional consistency.
- Schema changes: Modifying the schema of a large table can be time-consuming and requires careful planning.
</QuestionCard>

<QuestionCard id="bq-fund-09" number={9} question="What is the difference between a view and a materialized view in BigQuery?" difficulty="easy" topic="Views">
A materialized view in BigQuery is a precomputed table that stores the results of a query, while a view is a virtual table that derives its data from the underlying tables at query time.
</QuestionCard>

<QuestionCard id="bq-fund-10" number={10} question="What is the difference between a table decorator and a snapshot decorator in BigQuery?" difficulty="medium" topic="Backup & Recovery">

<QuestionCard id="bq-fund-11" number={11} question="How does BigQuery handle nested and repeated fields in JSON data?" difficulty="medium" topic="Fundamentals">
BigQuery supports nested and repeated fields in JSON data by flattening the structure and representing nested fields as separate columns. Repeated fields are represented as arrays in the flattened schema.
</QuestionCard>

<QuestionCard id="bq-fund-12" number={12} question="What is the purpose of the INFORMATION_SCHEMA in BigQuery?" difficulty="easy" topic="Schema">
The INFORMATION_SCHEMA in BigQuery is a virtual database schema that provides access to metadata about datasets, tables, views, columns, and other database objects. It allows users to query and retrieve information about the BigQuery resources.
</QuestionCard>

<QuestionCard id="bq-fund-13" number={13} question="How does BigQuery handle nested data types like arrays and structs?" difficulty="medium" topic="Fundamentals">
BigQuery supports nested data types like arrays and structs by allowing you to create tables with columns that contain nested fields. You can query and manipulate the nested data using dot notation or by using appropriate SQL functions.
</QuestionCard>

<QuestionCard id="bq-fund-14" number={14} question="Can you explain the concept of wildcard tables in BigQuery?" difficulty="medium" topic="Fundamentals">
Wildcard tables in BigQuery allow you to query multiple tables that match a specific pattern using a single query. They are useful when working with partitioned or date-sharded tables, enabling efficient querying of data across multiple tables.
</QuestionCard>

<QuestionCard id="bq-fund-15" number={15} question="Can you explain the concept of geographic data types in BigQuery?" difficulty="medium" topic="Fundamentals">
BigQuery supports geographic data types for representing spatial data, such as points, lines, and polygons. These types enable storage and querying of location-based information and provide functions for spatial analysis and calculations.

<QuestionCard id="bq-fund-16" number={16} question="How does BigQuery handle data consistency in distributed queries?" difficulty="medium" topic="Consistency">

<QuestionCard id="bq-fund-17" number={17} question="Can you explain the concept of BigQuery federated queries?" difficulty="medium" topic="Federated Queries">

## Common Interview Scenarios

- **"A dashboard query got expensive overnight — how do you investigate?"** Check bytes-scanned in the query plan, confirm the table is partitioned/clustered on the right columns, look for a `SELECT *` or missing partition filter, and check whether the query cache stopped applying because the underlying data changed.
- **"Design a schema for a high-volume event log table."** Partition by ingestion timestamp, cluster by a high-cardinality filter column like `user_id`, and use nested/repeated fields for event properties instead of a separate joined table.
- **"How would you keep near-real-time data queryable within seconds?"** Discuss the streaming API and streaming buffer, an `insertId`-based deduplication strategy, and the cost trade-off against micro-batch loading.
- **"Leadership wants predictable monthly BigQuery spend instead of surprise bills."** Cover flat-rate slot reservations versus on-demand pricing, budgets and alerts, and routine storage cleanup of unused tables and partitions.
