## Key Concepts

Redshift runs as an MPP (massively parallel processing) **cluster**: a **leader node** plans and coordinates queries while **compute nodes**, split into **slices**, store data and execute work in parallel. Two settings drive most performance conversations: the **distribution style/key**, which controls how rows are spread across slices (minimizing data movement during joins), and the **sort key**, which controls on-disk row order within each node (enabling range-restricted scans via zone maps). Data typically enters through the **COPY** command, which loads in parallel from S3 (or DynamoDB/EMR/JDBC), and leaves through **UNLOAD**; both have many tuning options (manifest files, compression, error handling). **VACUUM** reclaims space and re-sorts rows after deletes, while **ANALYZE** refreshes the statistics the query optimizer relies on. **Redshift Spectrum** extends queries directly onto data sitting in S3 without loading it, and **WLM (Workload Management)** governs concurrency and resource allocation across query queues. Redshift is built for OLAP, not OLTP — updates/deletes work but aren't its strength, and durability comes from automated snapshots to S3.

## Common Interview Scenarios

- **"A query against a large fact table has gotten slow — how do you approach it?"** Read the `EXPLAIN` plan for redistribution versus broadcast in joins, check whether the sort and distribution keys still match the query pattern, and confirm statistics are fresh (`ANALYZE`) and dead rows are reclaimed (`VACUUM`).
- **"Design a distribution and sort key strategy for a star schema fact table."** Distribute on the key most frequently joined against a large dimension (or use `ALL` for small dimensions), sort on the most common range filter such as a date column, and discuss the skew-versus-join-locality trade-off.
- **"How would you load a fresh batch of files from S3 efficiently and safely?"** Cover `COPY` with a manifest for parallel loading, `MAXERROR` for fault tolerance, staging-table-plus-atomic-swap patterns, and compression encoding.
- **"Leadership wants ad-hoc SQL against raw S3 data without duplicating storage."** Discuss Redshift Spectrum external tables versus Athena, and the cost/latency trade-offs of querying data in place.
