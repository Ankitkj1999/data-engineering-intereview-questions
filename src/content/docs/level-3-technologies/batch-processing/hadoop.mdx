## Hadoop: Distributed Processing Framework

**Hadoop is a framework for storing and processing very large datasets across clusters of commodity hardware — HDFS handles distributed storage and MapReduce/YARN handle distributed computation, with fault tolerance built in through replication rather than expensive hardware.**

## Key Concepts

Hadoop is built around two ideas: distribute the data, and move the computation to where the data already lives rather than moving data to the computation. **HDFS** breaks files into fixed-size blocks (64MB or 128MB) and replicates each block — three copies by default — across **DataNodes**, using rack-aware placement (two replicas on one rack, a third on a different rack) so that a single node or rack failure never loses data. A single **NameNode** holds all filesystem metadata (the directory tree and block locations) in memory; it never stores actual file data. DataNodes report their health back to the NameNode via periodic **heartbeats** and **block reports**, and a **Secondary/Checkpoint NameNode** periodically merges the edit log into the filesystem image so restarts stay fast — but it is *not* a hot standby, which made the NameNode a single point of failure in Hadoop 1.

**MapReduce** processes data in phases: an optional input split, a map phase (filter/transform), an optional combiner (local pre-aggregation to cut network traffic), a shuffle-and-sort phase (group by key), and a reduce phase (aggregate). In Hadoop 1 this was coordinated by a single **JobTracker** and per-node **TaskTrackers**; YARN later generalized resource management beyond just MapReduce. Across almost every question below, the recurring themes are: how fault tolerance is achieved through replication and heartbeats, why data locality drives performance, and where the classic single points of failure sit.

## Interview Questions

### HDFS Fundamentals & Architecture

Core concepts of the Hadoop Distributed File System — blocks, storage design, and how it differs from traditional and network-attached file systems.

<QuestionList topic="HDFS Fundamentals" storagePrefix="hadoop-hdfs">

<QuestionCard id="hadoop-hdfs-01" number={1} question="What are the main components of a Hadoop Application?" difficulty="easy" topic="HDFS">
A Hadoop application has four core components:
- **HDFS** — the distributed file system in which Hadoop data is stored, offering very high bandwidth.
- **Hadoop Common** — the common set of libraries and utilities used across Hadoop.
- **Hadoop MapReduce** — the algorithm used for large-scale data processing.
- **Hadoop YARN** — resource management for the cluster; it also schedules tasks for users.
</QuestionCard>

<QuestionCard id="hadoop-hdfs-02" number={2} question="What do you know about Block and Block scanner in HDFS?" difficulty="easy" topic="HDFS">
A large file in HDFS is broken into multiple parts, and each part is stored as a **Block**. By default, a block is 64MB.

A **Block Scanner** is a program every DataNode runs periodically to verify the checksum of every block it stores, in order to detect data corruption on that DataNode.
</QuestionCard>

<QuestionCard id="hadoop-hdfs-03" number={3} question="How will you disable a Block Scanner on HDFS DataNode?" difficulty="medium" topic="HDFS">
`hdfs-site.xml` has a configuration property `dfs.datanode.scan.period.hours` that controls how often, in hours, the Block Scanner runs. Setting it to `0` disables the Block Scanner entirely.
</QuestionCard>

<QuestionCard id="hadoop-hdfs-04" number={4} question="How can we update a file at an arbitrary location in HDFS?" difficulty="medium" topic="HDFS">
This is a trick question — HDFS does not allow updating a file at an arbitrary location. All files are written in append-only mode, meaning every write happens at the end of the file. There is no way to update a random location in an existing file.
</QuestionCard>

<QuestionCard id="hadoop-hdfs-05" number={5} question="What is the difference between NAS and DAS in Hadoop cluster?" difficulty="medium" topic="Storage">
- **NAS (Network Attached Storage)** — compute and storage are separated; storage is distributed over servers on a network.
- **DAS (Direct Attached Storage)** — storage is attached directly to the node doing the computation.

Hadoop is built on the principle of moving processing near the data, so it needs storage local to computation. DAS gives very good performance and can run on commodity hardware, making it more cost-effective. NAS is only preferable when very high bandwidth (around 10 GbE) is available.
</QuestionCard>

<QuestionCard id="hadoop-hdfs-06" number={6} question="How does indexing work in Hadoop?" difficulty="medium" topic="HDFS">
Hadoop indexing works at two levels:
- **Index based on File URI** — data is indexed by file; a search returns the files containing the data.
- **Index based on InputSplit** — data is indexed by the locations where the input split resides.
</QuestionCard>

<QuestionCard id="hadoop-hdfs-07" number={7} question="How will you check if a file exists in HDFS?" difficulty="easy" topic="HDFS">
Use the `hadoop fs -test` command, generally used for testing purposes:

```bash
hadoop fs -test -e -z -d file_uri
```

- `-e` checks existence of the file
- `-z` checks the file has non-zero size
- `-d` checks whether the path is a directory
</QuestionCard>

<QuestionCard id="hadoop-hdfs-08" number={8} question="How much storage is allocated by HDFS for storing a file of 25 MB size?" difficulty="medium" topic="Blocks">
HDFS stores all data in blocks, and the block size is configurable. With the default 64MB block size, storing a 25MB file still allocates at least one full block — so at least 64MB is allocated, even though only 25MB is actually used.
</QuestionCard>

<QuestionCard id="hadoop-hdfs-09" number={9} question="Why does HDFS store data in Block structure?" difficulty="medium" topic="Blocks">
Storing data in fixed-size blocks gives HDFS several benefits:
- **Fault tolerance** — replicating the same block across multiple locations means the data is still available even if one copy is inaccessible.
- **Large files** — files too large for a single disk can be split across blocks and stored across many machines.
- **Storage management** — fixed-size blocks make it easier to calculate storage usage and optimize algorithms that minimize data transfer across the network.
</QuestionCard>

<QuestionCard id="hadoop-hdfs-10" number={10} question="What is a Balancer in HDFS?" difficulty="medium" topic="HDFS">
Data in HDFS can end up unevenly spread across DataNodes — for example, right after a new DataNode is added to the cluster. The **Balancer** (sometimes called the Rebalancer) is an administrative tool that analyzes block placement and redistributes blocks to spread them more uniformly across DataNodes.
</QuestionCard>

<QuestionCard id="hadoop-hdfs-11" number={11} question="Why does the Hadoop Framework read a file in parallel but not write in parallel?" difficulty="hard" topic="HDFS I/O">
Reading is done in parallel because it lets Hadoop access data faster — the whole point of distributing storage.

Writing is *not* done in parallel because concurrent parallel writes to the same file would cause data inconsistency: if two nodes write to a file at the same time, neither knows what the other has written. For example, if 100MB of data (one 64MB block, one 36MB block) were written in parallel, the first block wouldn't know where the remaining data belongs. So Hadoop reads in parallel but writes sequentially.
</QuestionCard>

<QuestionCard id="hadoop-hdfs-12" number={12} question="If you change block size from 64MB to 128MB, what happens to existing data?" difficulty="medium" topic="Blocks">
Changing the block size does not affect data already written. After the change, every new file is chunked using the new 128MB block size, while existing data remains stored in its original 64MB chunks.
</QuestionCard>

<QuestionCard id="hadoop-hdfs-13" number={13} question="What is the maximum and minimum block size Hadoop allows?" difficulty="medium" topic="Blocks">
- **Minimum** — 512 bytes, which is the local OS file system's block size; you cannot go smaller.
- **Maximum** — there is no hard upper bound; it depends entirely on the environment/configuration.
</QuestionCard>

<QuestionCard id="hadoop-hdfs-14" number={14} question="What is NAS?" difficulty="easy" topic="Storage">
NAS (Network Attached Storage) is a file system where data resides on one centralized machine, and every cluster member reads and writes to that shared storage over the network. It is not as efficient as HDFS for Hadoop-style workloads.
</QuestionCard>

<QuestionCard id="hadoop-hdfs-15" number={15} question="How does HDFS differ from NAS?" difficulty="medium" topic="Storage">
- In **HDFS**, data blocks are distributed across the local drives of all machines in the cluster; in **NAS**, data sits on dedicated hardware.
- HDFS is designed to work with MapReduce, since computation moves to the data; NAS is not suited to this, since data is stored separately from computation.
- HDFS runs on a cluster of machines and provides redundancy via replication; NAS is served by a single machine and therefore doesn't provide the same data redundancy.
</QuestionCard>

<QuestionCard id="hadoop-hdfs-16" number={16} question="How is HDFS different from traditional file systems?" difficulty="medium" topic="HDFS">
HDFS is responsible for storing huge volumes of data across a cluster and is designed to run on commodity hardware. While it shares similarities with other distributed file systems, the differences matter:
- Highly fault-tolerant, designed for low-cost hardware.
- Provides high-throughput access to application data, suited to large data sets.
- Designed for applications that write data once and read it many times, requiring streaming-speed reads — "write-once, read-many" semantics.
</QuestionCard>

<QuestionCard id="hadoop-hdfs-17" number={17} question="What is HDFS block size, and how does it differ from a traditional file system's block size?" difficulty="medium" topic="Blocks">
In HDFS, data is split into blocks (typically 64MB or 128MB) distributed across the cluster, and each block is replicated multiple times (three, by default) across different nodes. HDFS stores each block as a separate file on the underlying local file system. Because of this scale and replication model, HDFS block size isn't directly comparable to a traditional file system's block size.
</QuestionCard>

<QuestionCard id="hadoop-hdfs-18" number={18} question="How does the client communicate with HDFS?" difficulty="medium" topic="HDFS">
Client applications use the Hadoop HDFS API. Whenever a client wants to locate, add, copy, move, or delete a file, it talks to the NameNode, which responds with the relevant DataNode servers where the data lives. Once the NameNode has provided that location, the client talks directly to the DataNode for the actual data.
</QuestionCard>

<QuestionCard id="hadoop-hdfs-19" number={19} question="Can you give some examples of Big Data?" difficulty="easy" topic="Big Data">
Real-world examples include: Facebook generating 500+ terabytes of data per day, the New York Stock Exchange generating about 1 terabyte of new trade data per day, and a jet airliner collecting 10 terabytes of sensor data for every 30 minutes of flight time.
</QuestionCard>

<QuestionCard id="hadoop-hdfs-20" number={20} question="What is the basic difference between traditional RDBMS and Hadoop?" difficulty="easy" topic="Big Data">
Traditional RDBMS is built for transactional systems — reporting and archiving data, and is useful when you want to look up a single record from a large data set. Hadoop is built to store huge volumes of data in a distributed file system and process it — useful when you want to pull in and analyze big data in one shot rather than seek a single record.
</QuestionCard>

<QuestionCard id="hadoop-hdfs-21" number={21} question="What is structured and unstructured data?" difficulty="easy" topic="Big Data">
- **Structured data** — easily identifiable, organized into a defined structure; the most common form is a database table of rows and columns.
- **Unstructured data** — anything not easily identifiable in that structured form, such as images, videos, documents, email, logs, and free text.
</QuestionCard>

<QuestionCard id="hadoop-hdfs-22" number={22} question="What is throughput, and how does HDFS get good throughput?" difficulty="medium" topic="HDFS">
Throughput is the amount of work done per unit of time — how fast data is accessed from the system, generally used as a performance measure. HDFS divides work and shares it among different systems, all executing their assigned tasks independently and in parallel, so the overall job completes in much less time. Reading data in parallel dramatically decreases actual read time, which is how HDFS achieves good throughput.
</QuestionCard>

<QuestionCard id="hadoop-hdfs-23" number={23} question="What is streaming access?" difficulty="medium" topic="HDFS">
HDFS follows a "write once, read many" model, which makes streaming access central to its design. HDFS focuses less on storing data and more on retrieving it at the fastest possible speed — especially useful for log analysis. Reading the complete data set matters more than the time to fetch any single record.
</QuestionCard>

<QuestionCard id="hadoop-hdfs-24" number={24} question="How is indexing done in HDFS?" difficulty="medium" topic="HDFS">
Hadoop has its own indexing approach. Depending on the configured block size, once data is stored, HDFS keeps track of the last part of the data, which points to where the next part is located. This chaining is the basis of how HDFS locates and reads data.
</QuestionCard>

<QuestionCard id="hadoop-hdfs-25" number={25} question="Who is a user in HDFS?" difficulty="easy" topic="HDFS">
A user is essentially anyone — a person — who has a query or needs some kind of data from the system.
</QuestionCard>

<QuestionCard id="hadoop-hdfs-26" number={26} question="Is the client the end user in HDFS?" difficulty="easy" topic="HDFS">
No. The client is an application that runs on a machine and is used to interact with the NameNode (job tracker) or DataNode (task tracker) on behalf of the end user.
</QuestionCard>

<QuestionCard id="hadoop-hdfs-27" number={27} question="What are the two types of writes in HDFS?" difficulty="medium" topic="HDFS">
- **Posted write** — write and forget, without waiting for acknowledgement (similar to traditional postal mail).
- **Non-posted write** — the writer waits for acknowledgement before proceeding (similar to a courier service).

Non-posted writes are naturally more expensive than posted writes, though both are asynchronous.
</QuestionCard>

<QuestionCard id="hadoop-hdfs-28" number={28} question="Why is reading done in parallel but not writing, in HDFS?" difficulty="medium" topic="HDFS I/O">
Reading is parallelized because it makes data access fast. Writing is not parallelized because writing to the same file from two nodes at once would create data inconsistency — neither node knows what the other has already written, so it becomes ambiguous which data should be stored and in what order.
</QuestionCard>

<QuestionCard id="hadoop-hdfs-29" number={29} question="Can Hadoop be compared to a NoSQL database like Cassandra?" difficulty="medium" topic="Big Data">
NoSQL is the closest technology to compare Hadoop against, but each has its own trade-offs. There is no distributed file system concept in most NoSQL databases. Hadoop itself is not a database — it's a distributed file system (HDFS) paired with a distributed programming framework (MapReduce).
</QuestionCard>

</QuestionList>

### NameNode, DataNode & Checkpointing

The roles of the master (NameNode) and slave (DataNode) daemons, and how metadata durability is protected via Secondary NameNode, Checkpoint Node, and Backup Node.

<QuestionList topic="NameNode & DataNode" storagePrefix="hadoop-nn">

<QuestionCard id="hadoop-nn-01" number={30} question="What is the difference between NameNode, Checkpoint Node, and Backup Node in HDFS?" difficulty="hard" topic="NameNode">
- **NameNode** — the heart of HDFS; it manages metadata. The actual file data isn't stored on the NameNode — instead it holds the directory tree of every file in the cluster. It uses two files for the namespace: the **fsimage** file (tracks the latest checkpoint of the namespace) and the **edits** file (a log of changes since the last checkpoint).
- **Checkpoint Node** — tracks the latest checkpoint in a directory structured like the NameNode's own. It creates checkpoints at regular intervals by downloading the edits and fsimage files from the NameNode, merging them locally, and uploading the merged image back to the active NameNode.
- **Backup Node** — provides checkpointing like the Checkpoint Node, but also maintains an up-to-date, in-memory copy of the file system namespace kept in sync with the active NameNode.
</QuestionCard>

<QuestionCard id="hadoop-nn-02" number={31} question="What are the two messages that NameNode receives from DataNode?" difficulty="easy" topic="NameNode">
- **Heartbeat** — signals that the DataNode is still alive; the NameNode relies on periodic heartbeats to decide whether to keep using a DataNode.
- **Block Report** — a list of all data blocks hosted on that DataNode, which tells the NameNode exactly what data lives where.
</QuestionCard>

<QuestionCard id="hadoop-nn-03" number={32} question="What data is stored in an HDFS NameNode?" difficulty="easy" topic="NameNode">
The NameNode is the central node of HDFS. It doesn't store any actual data on which MapReduce operations run — instead, it holds all the metadata about the data stored across DataNodes, including the directory tree of every file in the file system. It uses that metadata to manage everything stored on the different DataNodes.
</QuestionCard>

<QuestionCard id="hadoop-nn-04" number={33} question="What would happen if the NameNode crashed in an HDFS cluster?" difficulty="hard" topic="NameNode">
There's only one NameNode per HDFS cluster, and it holds all DataNode metadata — so it's a single point of failure. If it crashes, the system can become unavailable.

A **Secondary NameNode** can be configured to take periodic checkpoints of the file system, but it is *not* a backup of the NameNode. It can be used to help recreate and restart the NameNode after a crash, but it doesn't automatically fail over.
</QuestionCard>

<QuestionCard id="hadoop-nn-05" number={34} question="What are the main functions of the Secondary NameNode?" difficulty="medium" topic="NameNode">
- **FsImage** — stores a copy of the FsImage file and the EditLog.
- **NameNode crash recovery** — if the NameNode crashes, its FsImage copy can help recreate the NameNode.
- **Checkpoint** — runs periodic checkpoints to confirm HDFS data isn't corrupt.
- **Update** — periodically applies EditLog updates into the FsImage file, keeping it current and saving time during NameNode restarts.
</QuestionCard>

<QuestionCard id="hadoop-nn-06" number={35} question="What happens when the NameNode is down and a user submits a new job?" difficulty="medium" topic="NameNode">
Since the NameNode is the single point of failure in Hadoop, the job cannot execute — it fails while the NameNode is down. The user has to wait for the NameNode to restart and come back up before running the job.
</QuestionCard>

<QuestionCard id="hadoop-nn-07" number={36} question="What is a Checkpoint Node in HDFS?" difficulty="medium" topic="NameNode">
A Checkpoint Node periodically fetches the fsimage and edits from the NameNode, merges them, and creates a **checkpoint**, which it then uploads back to the NameNode. This differs from the Secondary NameNode, which also takes checkpoints but doesn't upload them back.

This matters most on NameNode restart: since the NameNode doesn't merge its edits into fsimage automatically at runtime, a long-running cluster can accumulate huge edit logs, making restarts slow because they have to be merged first. The Checkpoint Node offloads that merge work ahead of time, saving restart time.
</QuestionCard>

<QuestionCard id="hadoop-nn-08" number={37} question="What is a Backup Node in HDFS?" difficulty="medium" topic="NameNode">
Similar to a Checkpoint Node, a Backup Node takes a *stream* of edits from the NameNode (rather than reading them periodically), keeps them in memory, and also writes them to storage to create a new checkpoint — so it stays continuously in sync with the active NameNode. Unlike the Checkpoint Node, it does not upload any checkpoints back to the NameNode.
</QuestionCard>

<QuestionCard id="hadoop-nn-09" number={38} question="What is Safemode in HDFS?" difficulty="medium" topic="NameNode">
Safemode is the NameNode's read-only mode during startup — writes to the file system are not allowed while it's active. During this time, the NameNode collects data and statistics from all DataNodes, and only leaves Safemode once it has complete block information.

This avoids the NameNode prematurely deciding a block is under-replicated and starting replication before it actually knows about all existing replicas across the cluster. In Safemode, the NameNode first gathers replica counts across the cluster, then only creates new replicas where the count falls short of policy.
</QuestionCard>

<QuestionCard id="hadoop-nn-10" number={39} question="What is NameNode, and what are its responsibilities?" difficulty="easy" topic="NameNode">
NameNode is the logical daemon name for the master node — the heart of the entire Hadoop system. It stores metadata in the FsImage and gets all block information from DataNodes in the form of heartbeats.
</QuestionCard>

<QuestionCard id="hadoop-nn-11" number={40} question="What is a NameNode, and how many instances run on a Hadoop Cluster?" difficulty="medium" topic="NameNode">
The NameNode is the centerpiece of an HDFS file system — it keeps the directory tree of every file and tracks where the actual file data lives across the cluster, without storing that data itself.

There is only **one** NameNode process per Hadoop cluster, running on its own JVM, typically on a dedicated machine in production. It's the single point of failure for the HDFS cluster — when it goes down, the file system goes offline. Client applications talk to the NameNode whenever they want to locate, add, copy, move, or delete a file, and it responds with the relevant DataNode servers.
</QuestionCard>

<QuestionCard id="hadoop-nn-12" number={41} question="Is the NameNode also commodity hardware?" difficulty="easy" topic="NameNode">
No. The NameNode can never be commodity hardware, because the entire HDFS relies on it — it's the single point of failure in HDFS, so it needs to be a high-availability machine.
</QuestionCard>

<QuestionCard id="hadoop-nn-13" number={42} question="What is Metadata?" difficulty="easy" topic="NameNode">
Metadata is the information *about* the data stored on DataNodes — things like a file's location and size — rather than the data itself.
</QuestionCard>

<QuestionCard id="hadoop-nn-14" number={43} question="If a DataNode is full, how is it identified?" difficulty="medium" topic="DataNode">
When data is stored on a DataNode, the corresponding metadata is reported to the NameNode. The NameNode uses that metadata to identify when a DataNode is full.
</QuestionCard>

<QuestionCard id="hadoop-nn-15" number={44} question="If DataNodes increase, do we need to upgrade the NameNode?" difficulty="medium" topic="NameNode">
Usually not. NameNode sizing is determined by cluster size when the system is first installed, but since the NameNode only stores metadata (not the actual data), that requirement rarely arises even as DataNodes are added.
</QuestionCard>

<QuestionCard id="hadoop-nn-16" number={45} question="On what basis does the NameNode decide which DataNode to write on?" difficulty="medium" topic="NameNode">
Because the NameNode holds metadata about every DataNode, it knows which DataNode currently has free capacity, and uses that information to decide where to write.
</QuestionCard>

<QuestionCard id="hadoop-nn-17" number={46} question="What is the difference between Gen1 and Gen2 Hadoop with regards to the NameNode?" difficulty="hard" topic="NameNode">
In **Gen1** Hadoop, the NameNode is a single point of failure. In **Gen2** Hadoop, there's an Active/Passive NameNode structure — if the active NameNode fails, the passive NameNode takes over.
</QuestionCard>

<QuestionCard id="hadoop-nn-18" number={47} question="Do we require two servers for the NameNode and the DataNodes?" difficulty="medium" topic="NameNode">
Yes — the NameNode and DataNodes need different servers, because the NameNode requires a highly configurable, high-spec system since it stores metadata and location details for all files across all DataNodes, whereas DataNodes can run on lower-spec systems.
</QuestionCard>

<QuestionCard id="hadoop-nn-19" number={48} question="What will happen if a NameNode has no data?" difficulty="easy" topic="NameNode">
If a NameNode has no data (metadata), it can't function as a NameNode — in practical terms, a NameNode always needs to have some metadata to operate.
</QuestionCard>

</QuestionList>

### Replication, Rack Awareness & Fault Tolerance

How HDFS survives node and rack failures — replication factor, block placement policy, and heartbeat-driven failure detection.

<QuestionList topic="Fault Tolerance" storagePrefix="hadoop-fault">

<QuestionCard id="hadoop-fault-01" number={49} question="How will you get the distance between two nodes in Apache Hadoop?" difficulty="hard" topic="Rack Awareness">
Apache Hadoop provides `NetworkTopology.getDistance()` to get the distance between two nodes. The distance from a node to its parent is considered `1`.
</QuestionCard>

<QuestionCard id="hadoop-fault-02" number={50} question="What is Replication factor in HDFS?" difficulty="easy" topic="Replication">
Replication factor is the number of copies of a file HDFS maintains. An application can specify how many replicas it wants; this is stored by the NameNode. You can set it via the Hadoop `fs` shell:

```bash
# Set replication factor of a single file to 5
hadoop fs -setrep -w 5 /file_name

# Set replication factor for all files under a directory to 2
hadoop fs -setrep -w 2 /dir_name
```
</QuestionCard>

<QuestionCard id="hadoop-fault-03" number={51} question="What happens if an HDFS file has a replication factor of 1 and its DataNode crashes?" difficulty="medium" topic="Replication">
A replication factor of 1 means there is only a single copy of the file. If the DataNode holding that copy crashes, the data is lost with no way to recover it. Any business-critical data should always use a replication factor greater than 1.
</QuestionCard>

<QuestionCard id="hadoop-fault-04" number={52} question="What is the meaning of Rack Awareness in Hadoop?" difficulty="medium" topic="Rack Awareness">
Most Hadoop components — NameNode, DataNode, and others — are "rack-aware," meaning they know which rack they're physically on. This is central to implementing fault tolerance, since communication between nodes on the same rack is much faster than between nodes on different racks.

The NameNode maintains rack information for every DataNode and, when reading or writing data, tries to choose DataNodes that are close to each other for performance. Note that a replication factor of 3 doesn't mean computation happens three times — computation only runs on one copy of the file; if the node holding that copy doesn't respond, computation shifts to a second copy.
</QuestionCard>

<QuestionCard id="hadoop-fault-05" number={53} question="What is the default replication factor in HDFS?" difficulty="easy" topic="Replication">
The default replication factor in HDFS is **3** — meaning three copies of every block. It's configured via `dfs.replication` in `hdfs-site.xml`, and can also be set from the command line with the Hadoop `fs` command.
</QuestionCard>

<QuestionCard id="hadoop-fault-06" number={54} question="What does the NameNode consider before selecting a DataNode to place a data block?" difficulty="hard" topic="Rack Awareness">
The NameNode weighs a few factors when choosing DataNodes for block placement:
- It tries to keep at least one replica of a block on the same node that is writing it.
- It spreads different replicas of the same block across different racks, so a single rack failure doesn't lose all copies.
- One replica is kept on a different node on the *same* rack as the writer (distinct from keeping a copy on the writing node itself) to minimize network I/O.
- Overall, it tries to spread blocks uniformly across all DataNodes in the cluster.
</QuestionCard>

<QuestionCard id="hadoop-fault-07" number={55} question="How does a NameNode handle the failure of DataNodes?" difficulty="hard" topic="Fault Tolerance">
HDFS has a master/slave architecture: a single NameNode manages the file system namespace and regulates client access, while multiple DataNodes (usually one per cluster node) manage the storage attached to the machines they run on.

The NameNode periodically receives a Heartbeat and a Block Report from every DataNode. A Heartbeat implies the DataNode is functioning properly; a Block Report lists every block on that DataNode. If the NameNode doesn't receive a heartbeat from a DataNode within a set time window, it marks that DataNode dead. Since its blocks are now under-replicated, the system starts replicating them elsewhere — the NameNode orchestrates this, but the actual replication data transfer happens directly between DataNodes, never through the NameNode itself.
</QuestionCard>

<QuestionCard id="hadoop-fault-08" number={56} question="How are HDFS blocks replicated?" difficulty="hard" topic="Replication">
HDFS reliably stores very large files across a cluster by storing each file as a sequence of blocks — all blocks in a file are the same size except the last one. Blocks are replicated for fault tolerance, and both block size and replication factor are configurable per file (the replication factor can even be changed after file creation). Files in HDFS are write-once with exactly one writer at a time.

The NameNode makes all replication placement decisions using a rack-aware policy: by default, there are 3 total copies of each data block — 2 copies on DataNodes in the same rack, and the 3rd copy on a different rack.
</QuestionCard>

<QuestionCard id="hadoop-fault-09" number={57} question="Since data is replicated three times in HDFS, does that mean a calculation done on one node is also replicated on the other two?" difficulty="medium" topic="Replication">
No. With 3 replicas, when a MapReduce program runs, the calculation only executes against the original (or a chosen) copy of the data — the master node knows exactly which node has that data. Only if that node fails to respond does the calculation get redone against a second replica.
</QuestionCard>

<QuestionCard id="hadoop-fault-10" number={58} question="What is a Heartbeat in HDFS?" difficulty="easy" topic="Fault Tolerance">
A heartbeat is a signal indicating that a process is alive — a DataNode sends heartbeats to the NameNode, and a TaskTracker sends heartbeats to the JobTracker. If either the NameNode or the JobTracker stops receiving heartbeats, they conclude something's wrong with that DataNode or TaskTracker.
</QuestionCard>

<QuestionCard id="hadoop-fault-11" number={59} question="What is a Rack?" difficulty="easy" topic="Rack Awareness">
A Rack is a physical collection of DataNodes stored together at a single location — datanodes across the cluster can be physically located at different places, and a single location can contain multiple racks.
</QuestionCard>

<QuestionCard id="hadoop-fault-12" number={60} question="On what basis is data stored on a Rack?" difficulty="hard" topic="Rack Awareness">
When a client loads a file into the cluster, its content is divided into blocks. For every block, the client consults the NameNode and gets 3 DataNodes indicating where the block should be stored. The key rule: for every block of data, two copies exist on one rack, and a third copy exists on a different rack — this is the **Replica Placement Policy**.
</QuestionCard>

<QuestionCard id="hadoop-fault-13" number={61} question="Do the 2nd and 3rd replicas need to be placed in the same rack?" difficulty="medium" topic="Rack Awareness">
Yes — this is intentional, to avoid the risk of a single rack failure taking out multiple replicas of the same block.
</QuestionCard>

<QuestionCard id="hadoop-fault-14" number={62} question="What if the rack holding the second and third replicas fails along with a DataNode?" difficulty="medium" topic="Fault Tolerance">
If both that rack and a DataNode holding a replica in a different rack fail, there's no way to retrieve the data from that combination. To guard against this, you can increase the replication factor beyond the default of 3 so more copies exist across the cluster.
</QuestionCard>

<QuestionCard id="hadoop-fault-15" number={63} question="How does the NameNode handle DataNode failures?" difficulty="hard" topic="Fault Tolerance">
The NameNode periodically receives a Heartbeat and a Block Report from every DataNode. A Heartbeat implies the DataNode is functioning properly, and the Block Report lists every block that DataNode holds. When the NameNode notices it hasn't received a heartbeat from a DataNode after a set period, it marks that node dead. Since its blocks are now under-replicated, the system begins replicating them elsewhere — the NameNode orchestrates the process, but the actual data transfer happens directly between DataNodes, never passing through the NameNode.
</QuestionCard>

</QuestionList>

### MapReduce Fundamentals

The core MapReduce programming model — input splits, serialization, key/value contracts, data locality, and how a job's data flows through the framework.

<QuestionList topic="MapReduce Fundamentals" storagePrefix="hadoop-mr">

<QuestionCard id="hadoop-mr-01" number={64} question="What is the core concept behind the Apache Hadoop framework?" difficulty="easy" topic="MapReduce">
Apache Hadoop is built on the MapReduce algorithm, in which Map and Reduce operations process very large data sets: the **Map** method filters and sorts data, and the **Reduce** method summarizes it — a concept borrowed from functional programming. The key properties this gives Hadoop are scalability and fault tolerance, achieved through multi-threading and an efficient MapReduce implementation.
</QuestionCard>

<QuestionCard id="hadoop-mr-02" number={65} question="What is the use of the Context object in Hadoop?" difficulty="medium" topic="MapReduce">
Hadoop uses the Context object with the Mapper to interact with the rest of the system — it receives the system and job configuration in its constructor. The Context object is used to pass information through `setup()`, `cleanup()`, and `map()`, making key information available throughout the map operation.
</QuestionCard>

<QuestionCard id="hadoop-mr-03" number={66} question="What is the meaning of Data Locality in Hadoop?" difficulty="medium" topic="Data Locality">
Since Big Data volumes are huge, it doesn't make sense to move the data across the network. Instead, Hadoop moves the *computation* closer to the data, keeping data local to where it was stored and moving the compute tasks to the DataNodes that hold it.

Hadoop follows a preference order for locality:
- First, schedule the task on a node with the HDFS file on local disk.
- If not possible, schedule on a node on the same rack as the data.
- If that also isn't possible, schedule on a node with a copy of the data on a different rack.

This works well alongside the default replication factor of 3.
</QuestionCard>

<QuestionCard id="hadoop-mr-04" number={67} question="Why do we need Serialization in Hadoop MapReduce?" difficulty="medium" topic="MapReduce">
Since data may need to move between multiple DataNodes during map and reduce processing, Hadoop uses serialization to convert data from an in-memory object structure into a binary format for transfer, and deserialization to reliably convert it back into an object once it arrives.
</QuestionCard>

<QuestionCard id="hadoop-mr-05" number={68} question="Can you elaborate on a MapReduce job?" difficulty="medium" topic="MapReduce">
Based on configuration, a MapReduce job first splits the input data into independent chunks called blocks. These blocks are processed by the `map()` function first, then by the `reduce()` function. The framework itself takes care of sorting the map outputs and scheduling the tasks.
</QuestionCard>

<QuestionCard id="hadoop-mr-06" number={69} question="Why are compute nodes and storage nodes the same in Hadoop?" difficulty="medium" topic="Data Locality">
Compute nodes are the logical processing units, and storage nodes are the physical units holding data. By default, the Hadoop framework tries to minimize network traffic by following the **data locality** concept — code executes where the data is already stored, so the compute node and the DataNode end up being the same physical machine. As a result, Hadoop minimizes network wastage and processes data faster.
</QuestionCard>

<QuestionCard id="hadoop-mr-07" number={70} question="What is the importance of the configuration object in MapReduce?" difficulty="medium" topic="MapReduce">
The configuration object is used to set and get parameter name/value pairs, typically defined in an XML file. It's used to initialize values, read them from an external file, and set them as parameters at runtime. Parameter values set in the program always override values coming from external config files, and parameter values ultimately fall back to Hadoop's own defaults if nothing else is set.
</QuestionCard>

<QuestionCard id="hadoop-mr-08" number={71} question="Where is MapReduce not recommended?" difficulty="medium" topic="MapReduce">
MapReduce is not recommended for iterative processing — repeating output in a loop-like manner. It's also not well suited to a series of chained MapReduce jobs, since each job persists its data to local disk before the next job loads it again; that's a costly operation.
</QuestionCard>

<QuestionCard id="hadoop-mr-09" number={72} question="What methods control the Map and Reduce function's output types?" difficulty="medium" topic="MapReduce">
`setOutputKeyClass()` and `setOutputValueClass()` control the final output types. If the map output types differ from the final output types, they're set separately with `setMapOutputKeyClass()` and `setMapOutputValueClass()`.
</QuestionCard>

<QuestionCard id="hadoop-mr-10" number={73} question="What is InputSplit?" difficulty="medium" topic="InputSplit">
InputSplit is the chunk of data processed by a single mapper — in other words, the logical chunk of data assigned to one mapper. By default, `InputSplit` size equals block size.
</QuestionCard>

<QuestionCard id="hadoop-mr-11" number={74} question="How do you configure the split value?" difficulty="hard" topic="InputSplit">
By default, block size is 64MB, but the job tracker computes split size using these formulas:

```text
split size = min(max_splitsize, max(block_size, min_split_size))
split size = max(min_split_size, min(block_size, max_split_size))
```

By default, split size equals block size, and the number of splits always equals the number of mappers. Working through the formula with a 64MB block, a 512KB minimum split size, and a max split size dependent on environment (say 10GB):

```text
split size = min(max_splitsize, max(64, 512KB))
split size = min(10GB, 64) = 64MB

split size = max(min_split_size, min(block_size, max_split_size))
split size = max(512KB, min(64, 10GB))
split size = max(512KB, 64) = 64MB
```
</QuestionCard>

<QuestionCard id="hadoop-mr-12" number={75} question="What is the difference between block and split?" difficulty="easy" topic="Blocks">
- **Block** — how much data is physically stored together in HDFS.
- **Split** — how much data is processed together by a single mapper.
</QuestionCard>

<QuestionCard id="hadoop-mr-13" number={76} question="What is isSplitable()?" difficulty="medium" topic="InputSplit">
This defaults to `true`, and controls whether the input format is allowed to split the data. For unstructured data, splitting isn't usually recommended, so the entire file is processed as a single split — accomplished by setting `isSplitable()` to `false`.
</QuestionCard>

<QuestionCard id="hadoop-mr-14" number={77} question="What does a MapReduce job consist of?" difficulty="easy" topic="MapReduce">
A MapReduce job is a unit of work a client wants performed. It consists of the input data, the MapReduce program packaged as a JAR file, and configuration settings in XML files. Hadoop runs the job by dividing it into different tasks with the help of the JobTracker.
</QuestionCard>

<QuestionCard id="hadoop-mr-15" number={78} question="What is data locality?" difficulty="medium" topic="Data Locality">
Data locality means processing data wherever it already resides, rather than moving it — "moving computation is cheaper than moving data." This is only possible when the data is splittable, which it is by default.
</QuestionCard>

<QuestionCard id="hadoop-mr-16" number={79} question="What main configuration parameters must MapReduce programmers specify?" difficulty="medium" topic="MapReduce">

<QuestionCard id="hadoop-mr-17" number={80} question="What is RecordReader in a MapReduce job?" difficulty="medium" topic="MapReduce">

<QuestionCard id="hadoop-mr-18" number={81} question="What is OutputCommitter?" difficulty="hard" topic="MapReduce">

<QuestionCard id="hadoop-mr-20" number={83} question="What is SequenceFileInputFormat?" difficulty="medium" topic="MapReduce">

<QuestionCard id="hadoop-mr-21" number={84} question="Explain the input and output data format of the Hadoop Framework." difficulty="medium" topic="MapReduce">
The MapReduce framework operates exclusively on `(key, value)` pairs — it views the input to the job as a set of `(key, value)` pairs and produces a set of `(key, value)` pairs as output, possibly of different types than the input. The flow looks roughly like:

### Mapper & Reducer Internals

The lifecycle methods, interfaces, and behavior of Mappers and Reducers — from instantiation through `run()`, `setup()`, and `cleanup()`.

<QuestionList topic="Mapper & Reducer" storagePrefix="hadoop-mapred">

<QuestionCard id="hadoop-mapred-01" number={93} question="What are the core methods of a Reducer in Hadoop?" difficulty="medium" topic="Reducer">
The main task of a Reducer is to reduce a larger set of data sharing a key down to a smaller set. In Hadoop, a Reducer has three core methods:
- **`setup()`** — called at the start of a task to configure Reducer parameters.
- **`reduce()`** — the main operation; defines the task performed for each set of values sharing a key.
- **`cleanup()`** — called once `reduce()` is done, to clean up intermediate data or temp files.
</QuestionCard>

<QuestionCard id="hadoop-mapred-02" number={94} question="What are the primary phases of a Reducer in Hadoop?" difficulty="medium" topic="Reducer">
There are three primary phases:
- **Shuffle** — the Reducer copies the sorted output from each Mapper.
- **Sort** — the framework sorts the input to the Reducer by key, using merge sort. Shuffle and sort often happen concurrently.
- **Reduce** — output values associated with a key are reduced to produce the output result. Output from the Reducer is not re-sorted.

In short: **Shuffle, Sort, and Reduce.**
</QuestionCard>

<QuestionCard id="hadoop-mapred-03" number={95} question="When is a Reducer used?" difficulty="easy" topic="Reducer">
A Reducer is used to combine the output from multiple Mappers. It has 3 primary phases — sort, shuffle, and reduce. It's possible to process data without a reducer, but you need one whenever shuffle and sort are actually required.
</QuestionCard>

<QuestionCard id="hadoop-mapred-04" number={96} question="Where does the Shuffle and Sort process happen?" difficulty="medium" topic="Reducer">
After the Mapper generates output, the intermediate data is temporarily stored on the local file system — typically a temp directory configured in `core-site.xml`. The Hadoop Framework aggregates and sorts this intermediate data before handing it to the Reduce function. This temporary data is deleted from the local system once the job completes.
</QuestionCard>

<QuestionCard id="hadoop-mapred-05" number={97} question="What is the main difference between Mapper and Reducer?" difficulty="easy" topic="Mapper">
The `map()` method is called separately for each input key/value pair — it processes an input pair and emits intermediate key/value pairs. The `reduce()` method is called separately for each key and its associated list of values — it processes intermediate key/value pairs and emits the final key/value pairs. Both are initialized and called before any other method, and both take no additional parameters and produce no direct output beyond what they emit.
</QuestionCard>

<QuestionCard id="hadoop-mapred-06" number={98} question="What is the difference between map-side join and reduce-side join?" difficulty="hard" topic="Reducer">
- **Map-side join** — joining multiple tables entirely in the mapper. It requires a strict format and data that's already properly sorted; best when the datasets are smaller and can be handled without a reduce phase.
- **Reduce-side join** — joining multiple tables on the reducer side. Preferable when you have large tables — one very large and one relatively small — and need to properly partition the data. It's generally the best approach for joining sizable tables.
</QuestionCard>

<QuestionCard id="hadoop-mapred-07" number={99} question="What is a chain Mapper?" difficulty="medium" topic="Mapper">
A ChainMapper is a special class that lets multiple mapper classes run in a chained fashion within a single map task — one mapper's output feeds directly into the next mapper's input, letting you chain any number of mappers together.
</QuestionCard>

<QuestionCard id="hadoop-mapred-08" number={100} question="How do you do value-level comparison?" difficulty="medium" topic="MapReduce">
Hadoop can only compare at the key level — it does not support value-level comparison natively.
</QuestionCard>

<QuestionCard id="hadoop-mapred-09" number={101} question="What are the setup and cleanup methods used for?" difficulty="medium" topic="Mapper">
`setup()` and `cleanup()` solve the problem of not knowing where a processing block starts and ends. Each split gets one `setup()` and one `cleanup()` call, regardless of how many records/lines it contains — `setup()` initializes job resources, and `cleanup()` closes them once the last map for that split has finished. This significantly improves data transfer performance, since resources are opened once, used many times, and closed once, rather than repeatedly — the same idea applies on the Reducer side.
</QuestionCard>

<QuestionCard id="hadoop-mapred-10" number={102} question="What is Identity Mapper?" difficulty="easy" topic="Mapper">
Identity Mapper is the default Mapper class Hadoop uses when no other Mapper class is specified. It simply writes the input straight to output without performing any computation. The class is `org.apache.hadoop.mapred.lib.IdentityMapper`.
</QuestionCard>

<QuestionCard id="hadoop-mapred-11" number={103} question="What are the parameters of Mappers and Reducers?" difficulty="medium" topic="Mapper">
Mapper's four type parameters, in order:
- `LongWritable` (input key)
- `Text` (input value)
- `Text` (intermediate output key)
- `IntWritable` (intermediate output value)

Reducer's four type parameters, in order:
- `Text` (intermediate output key)
- `IntWritable` (intermediate output value)
- `Text` (final output key)
- `IntWritable` (final output value)
</QuestionCard>

<QuestionCard id="hadoop-mapred-12" number={104} question="How is a Mapper instantiated in a running job?" difficulty="medium" topic="Mapper">
The Mapper is instantiated within the running job itself, and it's passed a `MapContext` object that it can use to configure itself.
</QuestionCard>

<QuestionCard id="hadoop-mapred-13" number={105} question="Which methods are in the Mapper interface?" difficulty="medium" topic="Mapper">
Mapper has a `run()` method that calls its own `setup()` method exactly once, then calls `map()` once per input, and finally calls `cleanup()` once. All of these methods can be overridden.
</QuestionCard>

<QuestionCard id="hadoop-mapred-14" number={106} question="What happens if you don't override the Mapper methods?" difficulty="easy" topic="Mapper">
If you leave every method — including `map()` — unoverridden, it behaves as the identity function, emitting each input record as a separate output record unchanged.
</QuestionCard>

<QuestionCard id="hadoop-mapred-15" number={107} question="How can you add arbitrary key-value pairs in your Mapper?" difficulty="medium" topic="Mapper">
You can set arbitrary configuration data on your Job, for example with `Job.getConfiguration().set("myKey", "myVal")`, and then retrieve it in your mapper with `Context.getConfiguration().get("myKey")`. This is typically done in the Mapper's `setup()` method.
</QuestionCard>

<QuestionCard id="hadoop-mapred-16" number={108} question="How does Mapper's run() method work?" difficulty="medium" topic="Mapper">
`Mapper.run()` calls `map(KeyInType, ValInType, Context)` once for each key/value pair in the InputSplit assigned to that task.
</QuestionCard>

<QuestionCard id="hadoop-mapred-17" number={109} question="What is the Reducer used for?" difficulty="easy" topic="Reducer">
The Reducer reduces a set of intermediate values sharing a key down to a (usually smaller) set of values. The number of reducers for a job is set via `Job.setNumReduceTasks(int)`.
</QuestionCard>

<QuestionCard id="hadoop-mapred-18" number={110} question="Explain the core methods of the Reducer." difficulty="hard" topic="Reducer">
The Reducer API closely mirrors the Mapper's: a `run()` method receives a `Context` with the job's configuration plus interfacing methods to return data back to the framework. `run()` calls `setup()` once, `reduce()` once per key associated with the reduce task, and `cleanup()` once at the end. Any of these can be overridden with custom implementations — if none are overridden, the default operation is the identity function, passing values through unmodified.

The heart of the Reducer is `reduce()`, called once per key; its second argument is an `Iterable` returning all the values associated with that key.
</QuestionCard>

<QuestionCard id="hadoop-mapred-19" number={111} question="What are IdentityMapper and IdentityReducer in MapReduce?" difficulty="medium" topic="Mapper">
- **`org.apache.hadoop.mapred.lib.IdentityMapper`** — implements the identity function, mapping inputs directly to outputs. Used by default when the programmer doesn't set a Mapper class via `JobConf.setMapperClass`.
- **`org.apache.hadoop.mapred.lib.IdentityReducer`** — performs no reduction, writing every input value straight to output. Used by default when the programmer doesn't set a Reducer class via `JobConf.setReducerClass`.
</QuestionCard>

</QuestionList>

### Combiners, Partitioners & Shuffle

Local pre-aggregation with combiners, distributing reducer keys with partitioners, and how the shuffle/sort phases connect map output to reduce input.

<QuestionList topic="Combiners & Partitioners" storagePrefix="hadoop-part">

<QuestionCard id="hadoop-part-01" number={112} question="How does partitioning work in Hadoop?" difficulty="hard" topic="Partitioner">
Partitioning is the phase between Map and Reduce. Since the Partitioner's output feeds the Reducer, the number of partitions always equals the number of reducers.

The Partitioner splits Map-phase output into distinct partitions using a user-defined condition — often hash-based buckets. For example, to find the student with the maximum marks per gender per subject: Map first tags each row by gender; the Partitioner then splits each gender's rows by subject, sending each subject to its own Reducer, which finds the highest-scoring student within its partition.
</QuestionCard>

<QuestionCard id="hadoop-part-02" number={113} question="What is a Combiner in Hadoop?" difficulty="medium" topic="Combiner">
A Combiner is an optional step between Map and Reduce, also called a **Semi-Reducer**. It takes Map's output, creates key-value pairs, and passes summarized records with the same key on to the Reducer. By using a Combiner, you reduce the volume of data transferred between Mapper and Reducer — it does work similar to `reduce()`, but locally, on the Map machine itself.
</QuestionCard>

<QuestionCard id="hadoop-part-03" number={114} question="How will you create a custom Partitioner in a Hadoop job?" difficulty="hard" topic="Partitioner">
The Partition phase is optional and runs between Map and Reduce. To create a custom partitioner, extend `org.apache.hadoop.mapreduce.Partitioner` and override:

```java
getPartition(KEY key, VALUE value, int numPartitions)
```

`numPartitions` equals the number of reducers configured for the job. This method returns the partition number a given key/value record is assigned to — a corresponding reducer handles that partition and summarizes its data. Once the class is ready, register it on the job:

```java
job.setPartitionerClass(CustomPartitioner.class);
```
</QuestionCard>

<QuestionCard id="hadoop-part-04" number={115} question="What happens if the number of Reducers is 0?" difficulty="medium" topic="Reducer">
Setting the number of reducers to 0 is a valid configuration. In this scenario no reducer executes, so the mapper's output is treated as the final output, and Hadoop stores that in a separate folder.
</QuestionCard>

<QuestionCard id="hadoop-part-05" number={116} question="When do you reach for a Combiner?" difficulty="hard" topic="Combiner">
Mappers and reducers are independent — they don't talk to each other. When your aggregation function is **commutative** (`a . b = b . a`) and **associative** (`` a . (b . c) = (a . b) . c ``), you can safely use a Combiner to optimize the MapReduce process.

Many MapReduce jobs are bottlenecked by network bandwidth, so by default Hadoop tries to minimize data transfer. To that end, MapReduce lets you supply a user-defined Combiner function that runs directly on the map output — it's an optional optimization technique, not a required step.
</QuestionCard>

<QuestionCard id="hadoop-part-06" number={117} question="What is the main difference between a Combiner and a Reducer?" difficulty="medium" topic="Combiner">
Both are optional, but frequently used, in MapReduce. Three key differences:
- A **Combiner** only receives input from a single Mapper, while a **Reducer** receives input aggregated from multiple mappers.
- Use a Reducer when true aggregation is required; use a Combiner when the function is commutative (`a.b=b.a`) and associative (`a.(b.c)=(a.b).c`).
- A Combiner's input and output key/value types must match; a Reducer can accept any input type and produce any output type.
</QuestionCard>

<QuestionCard id="hadoop-part-07" number={118} question="What is Partition?" difficulty="medium" topic="Partitioner">
After the combiner and intermediate map output, the Partitioner controls the keys after sort and shuffle. It divides the intermediate data according to the number of reducers, so all the data within one partition is handled by a single reducer — each partition maps to exactly one reducer. Calling a reducer automatically triggers the partition step behind the scenes.
</QuestionCard>

<QuestionCard id="hadoop-part-08" number={119} question="What is the next step after the Mapper or map task?" difficulty="medium" topic="Partitioner">
The output of the Mapper is sorted, and partitions are created for it. The number of partitions depends on the number of reducers.
</QuestionCard>

<QuestionCard id="hadoop-part-09" number={120} question="How can we control which specific Reducer a key goes to?" difficulty="medium" topic="Partitioner">
By implementing a custom `Partitioner`, users can control exactly which keys (and hence records) end up at which Reducer.
</QuestionCard>

<QuestionCard id="hadoop-part-10" number={121} question="What is the use of a Combiner?" difficulty="medium" topic="Combiner">
A Combiner is an optional class, specified via `Job.setCombinerClass(ClassName)`, that performs local aggregation of intermediate outputs — cutting down the amount of data transferred from the Mapper to the Reducer.
</QuestionCard>

<QuestionCard id="hadoop-part-11" number={122} question="Explain the Shuffle phase." difficulty="medium" topic="Reducer">
Input to the Reducer is the sorted output of the mappers. In this phase, the framework fetches the relevant partition of each mapper's output, over HTTP.
</QuestionCard>

<QuestionCard id="hadoop-part-12" number={123} question="Explain the Reducer's sort phase." difficulty="medium" topic="Reducer">
The framework groups Reducer inputs by key, since different mappers might output the same key. The shuffle and sort phases occur simultaneously — as map outputs are fetched, they're merged, similar to a merge-sort.
</QuestionCard>

<QuestionCard id="hadoop-part-13" number={124} question="Explain the Reducer's reduce phase." difficulty="medium" topic="Reducer">
`reduce(MapOutKeyType, Iterable, Context)` is called for each pair in the grouped inputs. Output is typically written to the file system via `Context.write(ReduceOutKeyType, ReduceOutValType)`. Applications can use the Context to report progress, set status messages, and update counters — or simply signal they're still alive. Unlike Mapper output, Reducer output is not sorted.
</QuestionCard>

<QuestionCard id="hadoop-part-14" number={125} question="Is it possible for a job to have 0 Reducers?" difficulty="easy" topic="Reducer">
Yes — it's legal to set the number of reduce tasks to zero if no reduction is desired.
</QuestionCard>

<QuestionCard id="hadoop-part-15" number={126} question="What happens if the number of Reducers is 0?" difficulty="medium" topic="Reducer">
The output of the map tasks goes directly to the file system, into the output path set via `setOutputPath(Path)`. The framework does not sort the map outputs before writing them out.
</QuestionCard>

<QuestionCard id="hadoop-part-16" number={127} question="Can Reducers talk with each other?" difficulty="easy" topic="Reducer">
No — Reducers run in isolation.
</QuestionCard>

<QuestionCard id="hadoop-part-17" number={128} question="Where is the Mapper's intermediate data stored?" difficulty="medium" topic="Reducer">
The mapper's intermediate output is stored on the local file system (not HDFS) of each individual mapper node — typically a temporary directory location configured by the Hadoop administrator. This intermediate data is cleaned up once the job completes.
</QuestionCard>

<QuestionCard id="hadoop-part-18" number={129} question="Does the MapReduce programming model let Reducers communicate with each other?" difficulty="easy" topic="Reducer">
No — the MapReduce model doesn't allow reducers to communicate with each other. Reducers run in isolation.
</QuestionCard>

<QuestionCard id="hadoop-part-19" number={130} question="Can you set the number of Reducers to zero?" difficulty="medium" topic="Reducer">
Yes — setting reducers to zero is a valid configuration. With zero reducers, no reduce tasks run, and each mapper's output is stored as a separate file directly on HDFS. This differs from the non-zero case, where mapper output (intermediate data) is written to the *local* file system of each mapper's slave node rather than HDFS.
</QuestionCard>

</QuestionList>

### JobTracker, TaskTracker & Job Execution

The Hadoop 1 job execution model: how JobClient, JobTracker, and TaskTracker coordinate to run a job, schedule tasks, and recover from task failures.

<QuestionList topic="Job Execution" storagePrefix="hadoop-jt">

<QuestionCard id="hadoop-jt-01" number={131} question="What is the JobTracker's responsibility?" difficulty="medium" topic="JobTracker">
Scheduling a job's tasks onto the slave nodes, which execute them as directed. It also monitors tasks and re-executes any that fail.
</QuestionCard>

<QuestionCard id="hadoop-jt-02" number={132} question="What are the JobTracker and TaskTracker?" difficulty="medium" topic="JobTracker">
The MapReduce framework has a single JobTracker per cluster and one TaskTracker per node — since a cluster typically has multiple nodes, each cluster has one JobTracker and many TaskTrackers. The JobTracker schedules jobs and monitors TaskTrackers, re-executing tasks if a TaskTracker fails to run them. TaskTrackers follow the JobTracker's instructions, execute tasks, and — as a slave node — report job status back to the JobTracker via heartbeat.
</QuestionCard>

<QuestionCard id="hadoop-jt-03" number={133} question="Why does job scheduling matter in Hadoop MapReduce?" difficulty="medium" topic="JobTracker">
Scheduling is the systematic allocation of resources across multiple competing tasks. Since some procedures need to finish quickly and get priority, Hadoop supports several job schedulers. The default scheduler is **FIFO**; **Fair Scheduler** and **Capacity Scheduler** are the other popular options.
</QuestionCard>

<QuestionCard id="hadoop-jt-04" number={134} question="Can you elaborate on MapReduce Job architecture?" difficulty="hard" topic="JobTracker">
First, the programmer submits the MapReduce program to the `JobClient`. The `JobClient` requests a Job ID from the `JobTracker`, which returns one in the form `Job_HadoopStartedtime_00001` — a unique identifier.

Once the `JobClient` has the Job ID, it copies the job resources (`job.xml`, `job.jar`) to HDFS and submits the job to the `JobTracker`, which initiates and schedules it. Based on configuration, the job's input is split, and the `TaskTracker` retrieves the job resources from HDFS and launches a child JVM. Inside that child JVM, the map and reduce tasks run, and status is reported back to the `JobTracker`.
</QuestionCard>

<QuestionCard id="hadoop-jt-05" number={135} question="Why does TaskTracker launch a child JVM?" difficulty="medium" topic="TaskTracker">
Hadoop developers sometimes submit buggy or incorrect jobs. If the TaskTracker reused its own existing JVM, a bad task could interrupt the main JVM and affect other tasks. Running tasks in a separate child JVM means that if that child JVM tries to damage resources, TaskTracker can kill it and retry — or relaunch a fresh child JVM — without impacting the TaskTracker itself.
</QuestionCard>

<QuestionCard id="hadoop-jt-06" number={136} question="Why do JobClient and JobTracker submit job resources to the file system?" difficulty="medium" topic="JobTracker">
Because of data locality: moving computation is cheaper than moving data. The logic (the JAR file) and the splits need to end up where the data actually lives — on the DataNodes in the file system — so all job resources get copied to where the data is available.
</QuestionCard>

<QuestionCard id="hadoop-jt-07" number={137} question="How many Mappers and Reducers can run at once?" difficulty="medium" topic="JobTracker">
By default, Hadoop runs 2 mappers and 2 reducers per DataNode — each node has 2 map slots and 2 reduce slots. These defaults can be changed via the MapReduce configuration file.
</QuestionCard>

<QuestionCard id="hadoop-jt-08" number={138} question="What are the Job Resource files?" difficulty="easy" topic="JobTracker">
`job.xml` and `job.jar` are the core resources needed to process a job — the JobClient copies these resources to HDFS.
</QuestionCard>

<QuestionCard id="hadoop-jt-09" number={139} question="How many slots are allocated for each task?" difficulty="easy" topic="TaskTracker">
By default, each node has 2 slots for mapper tasks and 2 slots for reducer tasks — 4 slots total per node for processing data.
</QuestionCard>

<QuestionCard id="hadoop-jt-10" number={140} question="Why does TaskTracker launch a child JVM instead of using the existing JVM?" difficulty="medium" topic="TaskTracker">
Sometimes buggy code in a child thread can corrupt the parent thread, disrupting the entire MapReduce task because of a single programmer mistake. Launching a separate child JVM per mapper/task means TaskTracker can kill and relaunch that child if it misbehaves without harming the main JVM. TaskTracker typically retries a failed task up to 4 times.
</QuestionCard>

<QuestionCard id="hadoop-jt-11" number={141} question="Explain job scheduling through the JobTracker." difficulty="hard" topic="JobTracker">
The JobTracker communicates with the NameNode to identify data locations and submits work to the TaskTracker. The TaskTracker plays a critical role by notifying the JobTracker of any task failure — it's essentially the heartbeat reporter reassuring the JobTracker it's still alive. From there, the JobTracker decides what to do: it may resubmit the job elsewhere, mark a specific record unreliable, or blacklist a TaskTracker.
</QuestionCard>

<QuestionCard id="hadoop-jt-12" number={142} question="How many Maps are there in a particular job?" difficulty="hard" topic="JobTracker">
The number of maps is usually driven by the total size of the inputs — the total number of input file blocks. Generally you'll see around 10-100 maps per node, since task setup takes some time, so it's best if each map runs for at least a minute.

For example, 10TB of input with a 128MB block size would yield roughly 82,000 maps. `mapreduce.job.maps` can be used to hint at a desired count, but the actual number of tasks is ultimately controlled by however many splits `InputFormat.getSplits()` returns (a method you can override).
</QuestionCard>

<QuestionCard id="hadoop-jt-13" number={143} question="How many Reducers should be configured?" difficulty="hard" topic="JobTracker">
A good rule of thumb is:

```text
0.95 or 1.75 * (number_of_nodes * mapreduce.tasktracker.reduce.tasks.maximum)
```

At `0.95`, all reduces can launch immediately and start transferring map outputs as soon as maps finish. At `1.75`, the faster nodes finish their first wave of reduces and launch a second wave, doing much better load balancing. Increasing the number of reduces increases framework overhead, but improves load balancing and lowers the cost of individual task failures.
</QuestionCard>

<QuestionCard id="hadoop-jt-14" number={144} question="How many instances of JobTracker can run on a Hadoop Cluster?" difficulty="easy" topic="JobTracker">
Only one.
</QuestionCard>

<QuestionCard id="hadoop-jt-15" number={145} question="What is the JobTracker, and what does it perform in a Hadoop Cluster?" difficulty="hard" topic="JobTracker">
JobTracker is a daemon service that submits and tracks MapReduce tasks across the Hadoop cluster. It runs in its own JVM, typically on a dedicated machine, and each slave node is configured with the JobTracker's location. JobTracker is the single point of failure for the MapReduce service — if it goes down, all running jobs halt.

In a Hadoop cluster, JobTracker:
- Accepts jobs submitted by client applications.
- Talks to the NameNode to determine where the relevant data lives.
- Locates TaskTracker nodes with available slots at or near that data.
- Submits work to the chosen TaskTracker nodes.
- Gets notified by a TaskTracker when a task fails, and decides how to respond — resubmit elsewhere, mark the record unreliable, or blacklist the TaskTracker.
- Monitors TaskTrackers via heartbeat; if a TaskTracker's heartbeat stops arriving, it's deemed failed and its work is rescheduled elsewhere.
- Updates job status as work completes, which client applications can poll for.
</QuestionCard>

<QuestionCard id="hadoop-jt-16" number={146} question="How does a task get scheduled by a JobTracker?" difficulty="hard" topic="JobTracker">
TaskTrackers send heartbeat messages to the JobTracker every few minutes to reassure it they're still alive. These heartbeats also report the number of available slots, keeping the JobTracker up to date on where work can be delegated. When scheduling a task, the JobTracker first looks for an empty slot on the same server hosting the DataNode with the relevant data, and if none is available, looks for an empty slot on a machine in the same rack.
</QuestionCard>

<QuestionCard id="hadoop-jt-17" number={147} question="How many instances of TaskTracker run on a Hadoop Cluster?" difficulty="easy" topic="TaskTracker">
There is one TaskTracker daemon process for each slave node in the cluster.
</QuestionCard>

<QuestionCard id="hadoop-jt-18" number={148} question="How many maximum JVMs can run on a Slave Node?" difficulty="medium" topic="TaskTracker">
One or multiple Task Instances can run on each slave node, and each Task Instance runs as its own separate JVM process. The number of task instances is controllable via configuration — a high-end machine is typically configured to run more.
</QuestionCard>

<QuestionCard id="hadoop-jt-19" number={149} question="Are JobTracker and TaskTrackers present on separate machines?" difficulty="medium" topic="JobTracker">
Yes — JobTracker and TaskTracker run on different machines. This matters because JobTracker is the single point of failure for the MapReduce service; if it goes down, every running job halts.
</QuestionCard>

<QuestionCard id="hadoop-jt-20" number={150} question="What is a Task Tracker in Hadoop, and how many instances run on a cluster?" difficulty="hard" topic="TaskTracker">
A TaskTracker is a slave-node daemon that accepts tasks — Map, Reduce, and Shuffle operations — from a JobTracker. Only one TaskTracker process runs per Hadoop slave node, in its own JVM. Every TaskTracker is configured with a set of slots indicating how many tasks it can accept.

The TaskTracker starts separate JVM processes (Task Instances) to do the actual work, so a failed process doesn't take the TaskTracker itself down. It monitors these task instances, captures their output and exit codes, and notifies the JobTracker when they finish, successfully or not. It also sends heartbeats to the JobTracker every few minutes reporting its available slots.
</QuestionCard>

<QuestionCard id="hadoop-jt-21" number={151} question="What is a task instance in Hadoop, and where does it run?" difficulty="medium" topic="TaskTracker">
A task instance is the actual MapReduce job run on a slave node. The TaskTracker starts a separate JVM process for each — a Task Instance — so that a process failure doesn't take the TaskTracker down with it. Multiple task instances can run concurrently on a single node, depending on how many slots are configured; a new instance JVM spawns per task by default.
</QuestionCard>

<QuestionCard id="hadoop-jt-22" number={152} question="What does a typical Slave Node's configuration look like, and how many JVMs run on it?" difficulty="hard" topic="TaskTracker">
A single TaskTracker instance runs per slave node, as its own separate JVM process. A single DataNode daemon instance also runs per slave node, also as its own JVM. On top of those, one or multiple Task Instances run per node — each its own JVM — with the number configurable, and higher-end machines typically configured to run more.
</QuestionCard>

<QuestionCard id="hadoop-jt-23" number={153} question="If Reducers don't start before all Mappers finish, why does job progress show something like Map 50% / Reduce 10%?" difficulty="hard" topic="JobTracker">
Reducers start copying intermediate key-value pairs from mappers as soon as any are available — that data-transfer copying is what the progress percentage is tracking, so reduce progress shows up as soon as any intermediate output becomes available to transfer. However, the programmer-defined `reduce()` method itself is only invoked after every mapper has actually finished.
</QuestionCard>

<QuestionCard id="hadoop-jt-24" number={154} question="What happens to the JobTracker when the NameNode is down?" difficulty="medium" topic="JobTracker">
The NameNode holds all metadata and tracks DataNode failures via heartbeats, so when it's down, the entire cluster is effectively down — the NameNode is the single point of failure for a Hadoop installation, which takes MapReduce down with it.
</QuestionCard>

<QuestionCard id="hadoop-jt-25" number={155} question="What happens to the NameNode when the JobTracker is not up and running?" difficulty="medium" topic="JobTracker">
When the JobTracker is down, it stops functioning and every running job halts, since it's a single point of failure for MapReduce. Your MapReduce workloads are effectively down, but the NameNode is unaffected — HDFS remains accessible as long as the NameNode is working, even though you can't run a Hadoop job.
</QuestionCard>

</QuestionList>

### Hadoop Ecosystem, Deployment & Configuration

Deployment modes, cluster configuration files, hardware sizing, and ecosystem tools like Hadoop Streaming, DistCp, and Distributed Cache.

<QuestionList topic="Ecosystem & Configuration" storagePrefix="hadoop-eco">

<QuestionCard id="hadoop-eco-01" number={156} question="What is Hadoop Streaming?" difficulty="medium" topic="Ecosystem">
Hadoop Streaming is a Java utility, packaged as a JAR, that ships with the Hadoop distribution. It lets you create and run MapReduce jobs using an executable script — you write executable Mapper and Reducer scripts, pass them to Hadoop Streaming via a command, and the utility creates and submits the resulting Map/Reduce jobs to the cluster, letting you monitor them too.
</QuestionCard>

<QuestionCard id="hadoop-eco-02" number={157} question="What is the optimum hardware configuration to run Apache Hadoop?" difficulty="medium" topic="Cluster Hardware">
Recommended baseline: dual-core or dual-processor machines with 4GB or 8GB RAM using ECC (Error-Correcting Code) memory — without ECC, checksum errors become much more likely. For storage, high-capacity SATA drives (around 7200 RPM) work well, and roughly 10GB-bandwidth Ethernet networking is a good fit for a Hadoop cluster.
</QuestionCard>

<QuestionCard id="hadoop-eco-03" number={158} question="What are the default port numbers on which Nodes run in Hadoop?" difficulty="easy" topic="Config">
- NameNode runs on port `50070`
- Task Tracker runs on port `50060`
- Job Tracker runs on port `50030`
</QuestionCard>

<QuestionCard id="hadoop-eco-04" number={159} question="Why do we use commodity hardware in Hadoop?" difficulty="easy" topic="Cluster Hardware">
Hadoop doesn't require high-end servers with huge memory and processing power, so it can run on inexpensive systems with average RAM and processors — "commodity hardware." Since Hadoop MapReduce processes data in parallel, it's convenient to spread a task across many such servers and execute it there, which saves cost and is often faster than alternatives. Commodity hardware is also readily available, making scaling simpler — you just buy more of it, versus raising purchase orders for custom high-end machines built to order.
</QuestionCard>

<QuestionCard id="hadoop-eco-05" number={160} question="How does inter-cluster data copying work in Hadoop?" difficulty="medium" topic="Ecosystem">
Hadoop provides **DistCP** (Distributed Copy), a MapReduce-based utility for large inter- or intra-cluster copies of data. It creates Map tasks for the files given as input. After a DistCP copy, it's recommended to run crosschecks to confirm there's no data corruption and the copy completed fully.
</QuestionCard>

<QuestionCard id="hadoop-eco-06" number={161} question="What are the important configuration files in Hadoop?" difficulty="medium" topic="Config">
There are two categories of configuration files in a Hadoop cluster:
- **Default configuration** — `core-default.xml`, `hdfs-default.xml`, and `mapred-default.xml`, which specify the default cluster configuration. These are read-only.
- **Custom configuration** — site-specific files like `core-site.xml`, `hdfs-site.xml`, and `mapred-site.xml`, where you specify site-specific overrides.

Every job and HDFS operation uses the parameters defined across these files, letting you tune Hadoop for your use case. Hadoop's API has a `Configuration` class that loads these files and provides values to jobs at runtime.
</QuestionCard>

<QuestionCard id="hadoop-eco-07" number={162} question="What is the use of Distributed Cache in Hadoop?" difficulty="medium" topic="Ecosystem">
Distributed Cache is a Hadoop utility that improves job performance by caching files applications need. An application specifies which files to cache via `JobConf`, and the framework copies those files to the nodes where a task will execute, before the task starts. It supports both simple read-only text files and complex files like JARs and ZIPs.
</QuestionCard>

<QuestionCard id="hadoop-eco-08" number={163} question="How do you synchronize changes made to a file in the Distributed Cache?" difficulty="medium" topic="Ecosystem">
This is a trick question — Distributed Cache doesn't allow changes to a file at all. It's a mechanism for caching read-only data across nodes, so there's no way to update a cached file or synchronize changes to it.
</QuestionCard>

<QuestionCard id="hadoop-eco-09" number={164} question="Is Java mandatory to write MapReduce jobs?" difficulty="medium" topic="Ecosystem">
No. Hadoop itself is implemented in Java, but MapReduce applications don't have to be written in Java — Hadoop supports Python, Ruby, C++, and other languages. The Hadoop Streaming API lets you create and run Map/Reduce jobs with any executable or script as the mapper and/or reducer, and Hadoop Pipes lets programmers implement MapReduce applications in C++.
</QuestionCard>

<QuestionCard id="hadoop-eco-10" number={165} question="When do you reach for table partitioning in Hive?" difficulty="medium" topic="Ecosystem">
By default, Hive reads the entire dataset even when an application only needs a slice of it — a bottleneck for MapReduce-backed jobs. Hive's partitioning option addresses this: when creating a table, you partition it based on how it will typically be queried.
</QuestionCard>

<QuestionCard id="hadoop-eco-11" number={166} question="What are the important steps when partitioning a Hive table?" difficulty="hard" topic="Ecosystem">
Don't over-partition with too many tiny partitions — that adds overhead on the NameNode. If using dynamic partitioning, at least one static partition should exist, and strict mode should be enabled:

```sql
SET hive.exec.dynamic.partition = true;
SET hive.exec.dynamic.partition.mode = nonstrict;
```

Load data into a non-partitioned table first, then insert it into the partitioned table — you can't load data from local storage directly into a partitioned table:

```sql
INSERT OVERWRITE TABLE table_name PARTITION(year)
SELECT * FROM non_partition_table;
```
</QuestionCard>

<QuestionCard id="hadoop-eco-12" number={167} question="What is Commodity Hardware, and does it include RAM?" difficulty="easy" topic="Cluster Hardware">
Commodity hardware is an inexpensive system that isn't high-quality or high-availability — Hadoop can run on any average commodity hardware and doesn't need supercomputers or high-end machines. Yes, commodity hardware includes RAM, since some Hadoop services run in memory.
</QuestionCard>

<QuestionCard id="hadoop-eco-13" number={168} question="What is a Daemon?" difficulty="easy" topic="Ecosystem">
A daemon is a process or service that runs in the background. The term is common in Unix environments; the Windows equivalent is a "service," and the DOS equivalent is a "TSR."

<QuestionCard id="hadoop-eco-14" number={169} question="What are the three modes Hadoop can run in?" difficulty="medium" topic="Hadoop Modes">
- **Standalone (local) mode** — no Hadoop daemons running; everything runs in a single JVM.

This mode suits developers running MapReduce programs during development, and is the best environment for learning and debugging.

### Performance, Tuning & Troubleshooting

Memory sizing, speculative execution, and the operational commands and settings used to diagnose and administer a running cluster.

<QuestionList topic="Performance & Troubleshooting" storagePrefix="hadoop-perf">

<QuestionCard id="hadoop-perf-01" number={183} question="Why do we use the fsck command in HDFS?" difficulty="medium" topic="Troubleshooting">
`fsck` reports details on files and directories in HDFS. Its main uses:
- **`-delete`** — delete corrupted files in HDFS.
- **`-move`** — move corrupt files to lost+found.
- **`-locations`** — print all locations of a block in HDFS.
- **`-racks`** — print the network topology of DataNode locations.
- **`-blocks`** — print a report of blocks in HDFS.
</QuestionCard>

<QuestionCard id="hadoop-perf-02" number={184} question="How do you replace an HDFS data volume before shutting down a DataNode?" difficulty="hard" topic="Admin">
HDFS DataNodes support hot-swappable drives, which let you add or replace data volumes while the DataNode keeps running:
- Format and mount the new drive.
- Update the DataNode's `dfs.datanode.data.dir` configuration to reflect the new data volume directories.
- Run `dfsadmin -reconfig datanode HOST:PORT start` to kick off reconfiguration.
- Once reconfiguration completes, unmount the old data volume.
- Physically remove the old disks after unmounting.
</QuestionCard>

<QuestionCard id="hadoop-perf-03" number={185} question="How do you monitor memory used in a Hadoop cluster?" difficulty="hard" topic="Performance">
TaskTracker is the process most likely to consume high memory while running a task. You can configure TaskTracker to monitor the memory used by the tasks it launches, catching badly behaving tasks before they take the machine down with excessive memory use — including limiting maximum memory per task and per node. Relevant parameters include `mapred.cluster.map.memory.mb` / `mapred.cluster.reduce.memory.mb` (virtual memory size of a single map/reduce slot), `mapred.job.map.memory.mb` / `mapred.job.reduce.memory.mb` (default per-task memory limit), and `mapred.cluster.max.map.memory.mb` / `mapred.cluster.max.reduce.memory.mb` (the hard maximum per task).
</QuestionCard>

<QuestionCard id="hadoop-perf-04" number={186} question="How much RAM is required to process 64MB of data?" difficulty="hard" topic="Performance">
Assuming a 64MB block size with 2 mappers and 2 reducers: `64 * 4 = 256MB` of memory, and the OS needs at least 30% extra headroom, so `256 + 80 = 326MB` of RAM is roughly required to process that chunk of data. Processing unstructured data in practice tends to need even more memory than this baseline.
</QuestionCard>

<QuestionCard id="hadoop-perf-05" number={187} question="What is speculative execution?" difficulty="medium" topic="Performance">
Hadoop runs on commodity hardware, so individual machines can be slow or fail, which can delay an entire job waiting on one straggling machine. Speculative execution is a performance-optimization technique that runs the same logic on multiple systems simultaneously and uses whichever result finishes first — enabled by default.

For example, if the same logic runs on systems A, B, C, and D, finishing in 10, 8, 9, and 12 minutes respectively, the framework uses system B's result and kills the other three processes.
</QuestionCard>

<QuestionCard id="hadoop-perf-06" number={188} question="What is the meaning of Speculative Execution in Hadoop?" difficulty="medium" topic="Performance">
Speculative execution is a way of coping with uneven machine performance. In large clusters with hundreds or thousands of machines, some machines inevitably underperform, which can delay an entire job. To avoid this, Hadoop's speculative execution runs multiple copies of the same map or reduce task on different slave nodes, and uses the result from whichever node finishes first.
</QuestionCard>

<QuestionCard id="hadoop-perf-07" number={189} question="What is the communication channel between the client and NameNode/DataNode?" difficulty="medium" topic="Networking">
SSH is the mode of communication used between the client and the NameNode/DataNode.
</QuestionCard>

<QuestionCard id="hadoop-perf-08" number={190} question="What is a spill factor with respect to RAM?" difficulty="medium" topic="Performance">
Spill factor is the size threshold at which files start moving to a temp file — the `hadoop-temp` directory is used for this. The default value for `io.sort.spill.percent` is `0.80`; a value below `0.5` is not recommended.
</QuestionCard>

<QuestionCard id="hadoop-perf-09" number={191} question="How do you restart the NameNode?" difficulty="medium" topic="Admin">
Either run `stop-all.sh` followed by `start-all.sh`, or, more manually:

```bash
sudo hdfs

/etc/init.d/ha

```
</QuestionCard>

<QuestionCard id="hadoop-perf-10" number={192} question="How can you check whether the NameNode is working?" difficulty="easy" topic="Admin">
Use `/etc/init.d/hadoop-0.20-namenode status`, or more simply, run `jps`.
</QuestionCard>

<QuestionCard id="hadoop-perf-11" number={193} question="Why might you get a connection-refused Java exception running Hadoop fsck?" difficulty="medium" topic="Troubleshooting">
The most likely reason is that the NameNode isn't running on your VM.
</QuestionCard>

<QuestionCard id="hadoop-perf-12" number={194} question="What does /etc/init.d do?" difficulty="easy" topic="Admin">
`/etc/init.d` is where Linux daemons (services) live, and where you can check their status. This is a general Linux convention and has nothing Hadoop-specific about it.
</QuestionCard>

<QuestionCard id="hadoop-perf-13" number={195} question="How can you find the NameNode in the browser?" difficulty="easy" topic="Admin">
You don't use `localhost:8021` — the port to view the NameNode UI in the browser is `50070`.
</QuestionCard>

<QuestionCard id="hadoop-perf-14" number={196} question="What are the network requirements for Hadoop?" difficulty="medium" topic="Networking">
Hadoop core uses SSH to launch server processes on slave nodes, which requires password-less SSH connectivity between the master and all the slave and secondary machines.
</QuestionCard>

<QuestionCard id="hadoop-perf-15" number={197} question="Why do we need password-less SSH in a fully distributed environment?" difficulty="medium" topic="Networking">
In a live, fully distributed cluster, communication between nodes happens very frequently — the JobTracker needs to be able to send tasks to TaskTrackers quickly, without a password prompt slowing that down.
</QuestionCard>

<QuestionCard id="hadoop-perf-16" number={198} question="What does formatting the DFS mean?" difficulty="medium" topic="Admin">
Similar to formatting a drive in Windows, formatting the DFS structures data properly — but it also formats the NameNode in the process, which is generally not something you want to do casually.
</QuestionCard>

<QuestionCard id="hadoop-perf-17" number={199} question="Hadoop typically runs on Unix variants — can Microsoft Windows be used instead?" difficulty="easy" topic="Admin">
In practice, Ubuntu and Red Hat Linux are the best-supported operating systems for Hadoop. Windows can technically be used, but it's uncommon because of related support issues — frequent crashes and restarts make it unattractive for production. Windows isn't recommended as a Hadoop environment, though it's fine to experiment with for learning purposes.
</QuestionCard>

<QuestionCard id="hadoop-perf-18" number={200} question="What is the significance of SSH, and what port does it use?" difficulty="medium" topic="Networking">
SSH (Secure Shell) is a secure protocol and the most common way to administer remote servers safely — relatively simple and inexpensive to implement. A single SSH connection can host multiple channels and transfer data in both directions. SSH works on port `22` by default; it can be reconfigured to a different port, but that's not generally recommended. A password is required for local-host SSH for security whenever password-less communication hasn't been explicitly set up.
</QuestionCard>

<QuestionCard id="hadoop-perf-19" number={201} question="What is SSH, and how does it work between masters and slaves?" difficulty="medium" topic="Networking">
SSH (Secure Socket Shell) is a password-less, secure communication protocol that gives administrators a secure way to access a remote computer, with data packets sent securely across the network. It's used for communication both between masters and slaves and between any two hosts on a network generally. SSH first appeared in 1995 with SSH-1; SSH-2 is what's used today.
</QuestionCard>

## Common Interview Scenarios

- **"Design the storage layout for a 10TB file landing in HDFS."** Walk through block size (128MB), the resulting block count, the default replication factor of 3, and rack-aware placement (two replicas on one rack, a third on a different rack) — and explain why NameNode memory scales with block *count*, not data volume.
- **"The NameNode just crashed in production — what happens next?"** Cover the single-point-of-failure nature of Hadoop 1's NameNode, why the Secondary/Checkpoint NameNode is not a hot standby, and how Hadoop 2 introduced Active/Passive NameNode HA to remove this gap.
- **"A MapReduce job's reduce phase is the bottleneck — how do you speed it up?"** Discuss adding a Combiner to cut shuffle volume, tuning reducer count with the `0.95`/`1.75` rule of thumb, and using a custom Partitioner if a few keys are receiving disproportionate data (skew).
- **"Why does Hadoop move computation to data, instead of the other way around?"** Explain the data-locality principle, why network bandwidth was the scarce resource in early Hadoop clusters, and how the JobTracker/TaskTracker (or NameNode/DataNode) cooperate to schedule tasks near their data.
- **"A job fails intermittently on one slave node — how do you debug it?"** Talk through checking TaskTracker heartbeats and logs, whether speculative execution already worked around it by finishing on another node, and whether `fsck` reveals corrupted or under-replicated blocks tied to that node.
