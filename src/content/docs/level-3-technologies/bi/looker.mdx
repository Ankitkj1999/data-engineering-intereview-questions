## Looker: Business Intelligence & Exploration

**Looker is a modeling-first BI platform: instead of pulling data into its own engine, it defines reusable metrics and relationships in LookML and pushes queries down to the warehouse — which shapes almost every answer about performance, governance, and real-time capability.**

## Key Concepts

Looker's defining trait is that it doesn't store or move data — it's a semantic layer written in **LookML** that defines dimensions, measures, and relationships between tables, then compiles user interactions in an **Explore** into SQL run directly against the underlying warehouse. That "push-down" architecture explains most of Looker's trade-offs: it inherits the warehouse's scalability and real-time limits rather than imposing its own, relies on the warehouse (or **PDTs** — Persistent Derived Tables — materialized back into it) for performance, and depends on the warehouse's own security and partitioning features rather than reinventing them. Governance in Looker is mostly about **access control** (roles, Explore permissions, row-level filters) and **change management** (LookML in Git, Development Mode for safe iteration), since the modeling logic — not the data itself — is what Looker actually owns. Most interview questions boil down to: what does Looker do itself, and what does it delegate to the connected database?

## Interview Questions

### Fundamentals & Core Concepts

<QuestionList topic="Looker Fundamentals" storagePrefix="looker-fund">

<QuestionCard id="looker-fund-01" number={1} question="What is Looker?" difficulty="easy" topic="Fundamentals">
Looker is a business intelligence (BI) and data visualization platform that lets users explore, analyze, and share data insights, built around a modeling layer (LookML) rather than its own data engine.
</QuestionCard>

<QuestionCard id="looker-fund-02" number={2} question="How does Looker handle data integration?" difficulty="easy" topic="Fundamentals">
Looker connects to various data sources using database dialects and APIs, letting it integrate with different data platforms — SQL databases, cloud storage, and data warehouses — without moving the data into Looker itself.
</QuestionCard>

<QuestionCard id="looker-fund-03" number={3} question="Can you explain the concept of a LookML model in Looker?" difficulty="medium" topic="LookML">
LookML is Looker's modeling language. It defines the relationships between data tables and creates reusable dimensions and measures. A LookML model is a collection of files describing the structure and logic of your data — the single source of truth that every Explore and dashboard is built on top of.
</QuestionCard>

<QuestionCard id="looker-fund-04" number={4} question="What is a dimension in Looker?" difficulty="easy" topic="LookML">
A dimension represents a field or attribute in your data. It adds descriptive, qualitative context to your analysis — the counterpart to a measure, which holds the numeric values you aggregate.
</QuestionCard>

<QuestionCard id="looker-fund-05" number={5} question="How can you create a calculated field in Looker?" difficulty="medium" topic="LookML">
Calculated fields can be created either in LookML or directly in the Explore interface. LookML-based calculations support more advanced logic and are reusable across every Explore built on that model, while Explore-level calculations are quicker but scoped to a single query.
</QuestionCard>

<QuestionCard id="looker-fund-06" number={6} question="What is a Looker Explore?" difficulty="easy" topic="Explores">
A Looker Explore is the interface that lets users interactively explore and analyze data through a graphical, point-and-click UI, built on top of the relationships and fields defined in LookML.
</QuestionCard>

<QuestionCard id="looker-fund-07" number={7} question="What is the purpose of a Looker dashboard?" difficulty="easy" topic="Dashboards">
A Looker dashboard is a customizable collection of visualizations and reports that gives a consolidated view of key metrics and data insights, typically built from one or more Explores.
</QuestionCard>

<QuestionCard id="looker-fund-08" number={8} question="What is the purpose of a Looker filter?" difficulty="easy" topic="Explores">
A Looker filter lets users dynamically limit the data shown in a report or visualization based on specific criteria, without needing to rebuild the underlying query by hand.
</QuestionCard>

<QuestionCard id="looker-fund-09" number={9} question="What is the difference between Looker's Explore and join types?" difficulty="medium" topic="Explores">
An **Explore** defines a specific, queryable view of your data — the starting point and set of fields available to a user. **Join types** determine how the multiple underlying tables referenced by that Explore are combined (e.g., inner, left outer) to produce the consolidated result set.
</QuestionCard>

<QuestionCard id="looker-fund-10" number={10} question="Can you explain Looker's data modeling best practices?" difficulty="medium" topic="LookML">
- Use views and Explores deliberately — keep views focused on a single table/concept, and Explores focused on a specific analytical use case.
- Create reusable dimensions and measures rather than duplicating logic across models.
- Organize files and folders in LookML clearly, so models stay navigable as they grow.
- Apply consistent naming conventions across dimensions, measures, and Explores.
</QuestionCard>

<QuestionCard id="looker-fund-11" number={11} question="How does Looker handle data modeling for complex data structures?" difficulty="medium" topic="LookML">
LookML's flexibility handles complex data structures by letting you explicitly define relationships, joins, and transformations across files — so even deeply nested or multi-table structures can be modeled as a coherent set of Explores.
</QuestionCard>

<QuestionCard id="looker-fund-12" number={12} question="What are some advantages of using Looker compared to traditional BI tools?" difficulty="medium" topic="Fundamentals">
Looker offers a more agile, self-service approach to data analysis — users can explore data and build reports without heavy reliance on IT — while LookML provides centralized, powerful modeling capabilities that keep metric definitions consistent across the organization.
</QuestionCard>

<QuestionCard id="looker-fund-13" number={13} question="What are some limitations of Looker?" difficulty="medium" topic="Fundamentals">
Looker is fundamentally built around batch processing, so real-time capability is limited. It also depends entirely on the underlying data infrastructure (a data warehouse or database) rather than storing data itself, which means Looker's performance and scalability are only as good as what it's connected to.
</QuestionCard>

<QuestionCard id="looker-fund-14" number={14} question="Can Looker be integrated with other BI tools or data visualization platforms?" difficulty="medium" topic="Ecosystem">
Yes. Looker provides APIs and integrations that let it connect to other BI tools, visualization platforms, or custom applications, rather than functioning only as a closed system.
</QuestionCard>

<QuestionCard id="looker-fund-15" number={15} question="What are some advanced analytical capabilities offered by Looker?" difficulty="medium" topic="Fundamentals">
Advanced calculations, flexible data modeling, advanced filtering, derived tables (including PDTs), and support for custom visualizations for cases the built-in chart library doesn't cover.
</QuestionCard>

<QuestionCard id="looker-fund-16" number={16} question="Can Looker perform predictive analytics or machine learning?" difficulty="medium" topic="Fundamentals">
Not natively — Looker itself doesn't provide built-in predictive analytics or machine learning capabilities. It can, however, integrate with external tools or platforms that do, surfacing their output through Explores and dashboards.
</QuestionCard>

### LookML, PDTs & Development Workflow

<QuestionList topic="Looker LookML & PDTs" storagePrefix="looker-lookml">

<QuestionCard id="looker-lookml-01" number={17} question="Can you explain the concept of Looker PDTs (Persistent Derived Tables)?" difficulty="medium" topic="PDTs">
PDTs are precomputed tables that Looker builds back into the connected database to improve query performance. They're especially useful for complex or resource-intensive calculations that would otherwise be recomputed on every query.
</QuestionCard>

<QuestionCard id="looker-lookml-02" number={18} question="What is the purpose of Looker's PDT Events feature?" difficulty="medium" topic="PDTs">
PDT Events lets you schedule PDT builds at specific intervals, or trigger a rebuild based on specific events or upstream data changes, rather than relying purely on a fixed refresh schedule.
</QuestionCard>

<QuestionCard id="looker-lookml-03" number={19} question="What is the purpose of Looker's PDT Performance Optimization features?" difficulty="hard" topic="PDTs">
These features let you manage PDT builds and refresh schedules more efficiently, so PDTs improve query performance without themselves becoming a source of unnecessary database load.
</QuestionCard>

<QuestionCard id="looker-lookml-04" number={20} question="What is the purpose of Looker's Liquid syntax in LookML?" difficulty="hard" topic="LookML">
Liquid syntax is used inside LookML to dynamically generate SQL, or to control the behavior of dimensions, measures, or filters based on conditions or user inputs — effectively templating logic that would otherwise require duplicating LookML.
</QuestionCard>

<QuestionCard id="looker-lookml-05" number={21} question="What is the purpose of Looker's Development Mode?" difficulty="medium" topic="Administration">
Development Mode lets developers change LookML models without affecting the production environment, giving them a safe space to test changes before deploying them live.
</QuestionCard>

<QuestionCard id="looker-lookml-06" number={22} question="How can you create a custom visualization in Looker?" difficulty="medium" topic="Visualization">
Looker supports custom visualizations built with custom JavaScript or HTML — you can build your own from scratch or adapt third-party visualization libraries for cases the built-in chart types don't cover.
</QuestionCard>

<QuestionCard id="looker-lookml-07" number={23} question="What is the purpose of Looker's SQL Runner?" difficulty="medium" topic="SQL Runner">
SQL Runner provides a SQL interface directly inside Looker, letting users write and execute custom SQL queries against connected data sources without going through LookML or an Explore.
</QuestionCard>

<QuestionCard id="looker-lookml-08" number={24} question={"What is the purpose of Looker's \"Always Use SQL Runner\" option?"} difficulty="medium" topic="SQL Runner">
This option lets users bypass LookML entirely and directly write/execute custom SQL through the SQL Runner interface — useful for ad-hoc investigation, at the cost of losing LookML's reusable modeling layer for that query.
</QuestionCard>

<QuestionCard id="looker-lookml-09" number={25} question="Can you explain Looker's dynamic time frame filters?" difficulty="medium" topic="Explores">
Dynamic time frame filters let users select relative, time-based periods — such as "last month" or "last year" — that are evaluated dynamically based on the current date, rather than a fixed date range.
</QuestionCard>

<QuestionCard id="looker-lookml-10" number={26} question="How does Looker handle data versioning and change management for LookML models?" difficulty="medium" topic="Version Control">
Looker integrates with Git (or other version control systems) so LookML models can be version-controlled like any other codebase. This lets teams track changes, compare versions, revert to a previous version of a model, and manage changes through a standard review process rather than editing production models directly.
</QuestionCard>

<QuestionCard id="looker-lookml-11" number={27} question="What is Looker's Explore API used for?" difficulty="medium" topic="API">
The Explore API lets developers programmatically query and retrieve data from Looker Explores, enabling integration with external applications that need Looker-modeled data without going through the UI.
</QuestionCard>

<QuestionCard id="looker-lookml-12" number={28} question="What is Looker's Explore Embed feature used for?" difficulty="medium" topic="Embedding">
Explore Embed lets developers embed Looker's Explore functionality directly into external applications, giving end users a full data-exploration experience inside a product they're already using rather than a separate Looker login.
</QuestionCard>

<QuestionCard id="looker-lookml-13" number={29} question="What are the advantages of using Looker's embedded analytics feature?" difficulty="medium" topic="Embedding">
Embedded analytics lets you embed Looker dashboards and reports directly into external applications, giving users seamless access to data insights inside your own product rather than requiring them to switch to a separate BI tool.
</QuestionCard>

</QuestionList>

## Common Interview Scenarios

- **"A dashboard is slow — where do you look first?"** Reason through whether the underlying SQL is inefficient, whether a PDT should be materializing the expensive part, whether caching is configured sensibly, and whether the bottleneck is actually the warehouse rather than Looker itself.
- **"The business wants near-real-time dashboards — is that realistic in Looker?"** Explain Looker's batch-first architecture, and how frequent refresh schedules or a streaming-fed warehouse table can get close without claiming true real-time.
- **"Two teams need to see the same metric defined identically — how do you enforce that in Looker?"** Talk through centralizing the definition as a LookML measure, Git-managed model changes, and Development Mode so nobody edits the production definition directly.
- **"How would you restrict a reseller's users to only their own data in a shared Looker instance?"** Cover Explore permissions, row-level access controls, and the multi-tenant access-control model.
