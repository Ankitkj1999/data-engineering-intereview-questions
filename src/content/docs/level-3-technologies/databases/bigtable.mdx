## Bigtable: Google's Wide-Column NoSQL Database

**Google Cloud Bigtable is a fully managed, wide-column NoSQL database built for massive, low-latency workloads — its interview questions center on how row-key design drives performance, since Bigtable provides no secondary indexes, no automatic query optimization, and no joins to fall back on.**

## Key Concepts

Bigtable's data model is a sparse, distributed, **multi-dimensional sorted map**: every cell is addressed by a **row key**, a **column key** (grouped into column families), and a **timestamp** for versioning. Because rows are stored in row-key sort order and Bigtable has **no secondary indexes** and **no automatic query optimization**, the row key is the single most consequential design decision — it determines both read efficiency (range scans work only along the key) and write distribution (poorly chosen keys, especially monotonically increasing ones, create **hotspots**). Architecturally, a table is split into **tablets**, each served by exactly one **tablet server**, with a **master server** handling tablet assignment and load balancing while actual bytes live on **Colossus**, Google's distributed file system. Writes land in an in-memory **memstore** before flushing to immutable sorted files, and **compaction** periodically merges those files while **Bloom filters** skip unnecessary disk reads. Bigtable offers **no ACID transactions** beyond atomic single-row read-modify-write operations, replicates across clusters with **eventual consistency** and last-writer-wins conflict resolution, and delegates access control to **Cloud IAM**.

- **"Design a row key for a time-series IoT sensor table."** Explain why a naive `device_id + timestamp` key hotspots on write, then discuss salting, key reversal, or `device_id#reverse_timestamp` bucketing to spread writes while preserving efficient range scans.
- **"Reads on one tablet are slow while the rest of the cluster is idle — what's happening?"** Diagnose a hotspot caused by a poorly distributed row key, and walk through tablet splitting and a row-key redesign as the fix.
- **"Compare Bigtable to Cassandra or HBase for a new project."** Cover managed-service versus self-hosted operations, HBase API compatibility, differing replication/consistency models, and the trade-off of GCP ecosystem lock-in.
- **"How would you keep an application read-consistent across a multi-region Bigtable replication setup?"** Discuss eventual consistency, last-writer-wins conflict resolution, and routing reads to the nearest replica versus what to do if the application actually needs strong consistency.
