## Delta Lake: Reliable Data Storage

**Delta Lake is an open-source storage layer that brings ACID transactions, schema enforcement, and time travel to Parquet-based data lakes — turning a folder of files into something that behaves like a table.**

## Key Concepts

Delta Lake sits on top of **Parquet** and adds a **transaction log** (`_delta_log/`) that records every change to a table, which is what makes **ACID transactions**, **time travel**, and reliable **schema enforcement/evolution** possible. Writers use **optimistic concurrency control** against that log, so multiple concurrent writers can commit safely, with conflicts detected and retried rather than silently corrupting data. Table maintenance revolves around `OPTIMIZE` (compacting small files and applying **Z-order clustering**) and `VACUUM` (removing old, unreferenced files) — both important because frequent writes or streaming ingestion tend to produce many small files. Delta also supports row-level `MERGE`/`DELETE`/`UPDATE`, **Change Data Capture (CDC)** for downstream consumers, and performance features like **data skipping** built on file-level statistics.

## Common Interview Scenarios

- **"How does Delta Lake guarantee ACID transactions on top of object storage, which has no native transactions?"** Explain the role of the `_delta_log` transaction log and optimistic concurrency control — writers propose new log entries and retry on conflict, rather than relying on storage-level locking.
- **"A Delta table has thousands of small files after months of streaming writes — how do you fix it?"** Talk through `OPTIMIZE` for compaction (and Z-order clustering on frequently filtered columns), followed by `VACUUM` to reclaim space, and how you'd schedule that maintenance.
- **"How would you implement an upsert pipeline into a Delta table?"** Walk through `MERGE INTO` with match/no-match conditions, and how CDC (`delta.enableChangeDataFeed`) could let downstream consumers react to just the changed rows instead of rescanning the table.
