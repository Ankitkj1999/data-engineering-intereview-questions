## Key Concepts

A data warehouse exists to answer one question well: how do you turn scattered, write-optimized operational data into a structure that's fast and reliable to query for analytics? The foundational split is **OLTP vs. OLAP** — OLTP systems (the applications generating your data) are normalized and optimized for many small, concurrent writes; OLAP systems are denormalized and optimized for large, read-heavy aggregations across history. Warehouses exist because running analytical queries directly against an OLTP system either degrades it or gives you an incomplete, historyless answer.

Getting data from source to warehouse is the job of **ETL/ELT** — extract from source systems, transform (clean, conform, apply business rules), and load into the warehouse, or in the ELT variant, load raw and transform inside the warehouse using its own compute. Once data lands, **dimensional modeling** — star and snowflake schemas, fact/dimension tables, slowly changing dimensions — is the standard way warehouses organize data for query performance and business usability; see the [Data Modeling](/level-2-core-concepts/data-modeling/) page for schema-design specifics, since this page stays at the platform and architecture layer around it.

The landscape has also evolved beyond a single "data warehouse" box: **data lakes** store raw, schema-on-read data cheaply at scale, and **lakehouses** try to combine lake economics with warehouse-style transactions and performance. Vendor-specific implementation details for Snowflake, BigQuery, Redshift, and Databricks live on their own pages — this page stays at the concepts and trade-offs level that shows up across all of them.

<QuestionCard id="dwh-fund-07" number={7} question="What is Data Warehousing Hierarchy?" difficulty="hard" topic="Dimensions">
Hierarchies are logical structures that use ordered levels as a means of organizing data. A hierarchy can be used to define data aggregation. For example, in a time dimension, a hierarchy might aggregate data from the month level to the quarter level to the year level. A hierarchy can also be used to define a navigational drill path and to establish a family structure. Within a hierarchy, each level is logically connected to the levels above and below it. Data values at lower levels aggregate into the data values at higher levels. A dimension can be composed of more than one hierarchy. For example, in the product dimension, there might be two hierarchies--one for product categories and one for product suppliers. Dimension hierarchies also group levels from general to granular. Query tools use hierarchies to enable you to drill down into your data to view different levels of granularity. This is one of the key benefits of a data warehouse. When designing hierarchies, you must consider the relationships in business structures. Hierarchies impose a family structure on dimension values. For a particular level value, a value at the next higher level is its parent, and values at the next lower level are its children. These familial relationships enable analysts to access data quickly.
</QuestionCard>

<QuestionCard id="dwh-schema-05" number={19} question="What is the difference between star and snowflake schema, and when is each better?" difficulty="medium" topic="Star Schema">
- **Star schema** — dimension tables are mapped directly around one or more fact tables. It's a renormalized (denormalized) model, so no complicated joins are needed and queries run fast. It's the default choice when raw query performance matters most.
- **Snowflake schema** — the normalized form of a star schema: dimension tables are split into multiple related tables, which reduces redundancy and makes direct modification of dimension data easier, at the cost of more, more complex joins and somewhat slower queries.

In short: pick snowflake when you have genuine one-to-many hierarchies within a dimension and want easier maintenance/browsing; pick star when query performance and simplicity are the priority, which is most analytics/BI workloads.
</QuestionCard>

<QuestionCard id="dwh-schema-09" number={23} question="What is Surrogate Key and where we use it?" difficulty="hard" topic="Surrogate Keys">
Surrogate key is a substitution for the natural primary key. It is just a unique identifier or number for each row that can be used for the primary key to the table. The only requirement for a surrogate primary key is that it is unique for each row in the table.
Data warehouses typically use a surrogate, (also known as artificial or identity key), key for the dimension tables primary keys. They can use Info sequence generator, or Oracle sequence, or SQL Server Identity values for the surrogate key.
It is useful because the natural primary key (i.e. Customer Number in Customer table) can change and this makes updates more difficult.
Some tables have columns such as AIRPORT_NAME OR CITY_NAME which are stated as the primary keys (according to the business users) but ,not only can these change, indexing on a numerical value is probably better and you could consider creating a surrogate key called, say, AIRPORT_ID. This would be internal to the system and as far as the client is concerned, you may display only the AIRPORT_NAME.
</QuestionCard>

<QuestionCard id="dwh-oltp-01" number={59} question="Differentiate between OLTP and OLAP?" difficulty="medium" topic="OLAP">
OLTP stands for Online Transaction Process System

### Normalization & Relational Theory

<QuestionList topic="Normalization & Relational Theory" storagePrefix="dwh-norm">

<QuestionCard id="dwh-norm-01" number={123} question="What is Normalization?" difficulty="medium" topic="Normalization">
Database normalization is a data design and organization process applied to data structures based on rules that help build relational databases. In relational database design, the process of organizing data to minimize redundancy. Normalization usually involves dividing a database into two or more tables and defining relationships between the tables. The objective is to isolate data so that additions, deletions, and modifications of a field can be made in just one table and then propagated through the rest of the database via the defined relationships.
</QuestionCard>

<QuestionCard id="dwh-norm-02" number={124} question="What are different Normalization Forms?" difficulty="hard" topic="Normalization">
- 1NF: Eliminate Repeating Groups Make a separate table for each set of related attributes, and give each table a primary key. Each field contains at most one value from its attribute domain.
- 2NF: Eliminate Redundant Data If an attribute depends on only part of a multi-valued key, remove it to a separate table.
- 3NF: Eliminate Columns Not Dependent On Key If attributes do not contribute to a description of the key, remove them to a separate table. All attributes must be directly dependent on the primary key.
- BCNF: Boyce-Codd Normal Form If there are non-trivial dependencies between candidate key attributes, separate them out into distinct tables.
- 4NF: Isolate Independent Multiple Relationships No table may contain two or more 1:n or n:m relationships that are not directly related.
- 5NF: Isolate Semantically Related Multiple Relationships There may be practical constrains on information that justify separating logically related many-to-many relationships.
- ONF: Optimal Normal Form A model limited to only simple (elemental) facts, as expressed in Object Role Model notation.
- DKNF: Domain-Key Normal Form A model free from all modification anomalies. Remember, these normalization guidelines are cumulative. For a database to be in 3NF, it must first fulfill all the criteria of a 2NF and 1NF database.
</QuestionCard>

## Common Interview Scenarios

- **"Design a data warehouse for a retail company's sales reporting."** Start by nailing down the source systems and grain, then walk through staging, a star schema for the sales fact table (product, store, date, promotion dimensions), and how OLTP data gets denormalized for reporting performance.
- **"Our OLAP queries are slow — where do you start?"** Probe fact table grain, aggregation/rollup strategy, indexing choice (bitmap vs. B-tree), and whether the underlying MOLAP/ROLAP/HOLAP trade-offs actually fit the data volume and query patterns.
- **"How would you choose between a data warehouse, a data lake, and a lakehouse for a new analytics platform?"** Talk through schema-on-write vs. schema-on-read, cost profile, data variety, and the query patterns each is actually good at.
- **"A dimension needs to track history — how do you model that, and how does it flow through the ETL?"** Walk through SCD Type 2 (or a hybrid approach) plus the load logic — surrogate keys, effective-dating, detecting changes — that keeps it correct.
- **"How would you migrate an on-prem warehouse to the cloud without breaking downstream BI reports?"** Discuss re-architecting ETL/ELT, keeping schema parity during the transition, and a reconciliation/validation strategy before cutting reports over.
