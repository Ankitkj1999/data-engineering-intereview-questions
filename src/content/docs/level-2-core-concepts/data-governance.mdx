## Data Governance: Ownership, Access & Compliance

**Data governance is the set of processes, policies, and roles that make data discoverable, trusted, secure, and compliant as a platform scales beyond a single team.**

## Key Concepts

Governance starts with clear ownership: a **data owner** is accountable for a dataset's business meaning, a **data steward** maintains its quality and metadata day to day, and a **data custodian** operates the underlying platform. On top of ownership sits access control — **RBAC** grants permissions to roles, while **ABAC** evaluates dynamic attributes (department, region, sensitivity) for more flexible but harder-to-manage policies — plus finer-grained row- and column-level security for multi-tenant and PII-heavy datasets. A **data catalog** ties it together by making datasets discoverable, with lineage, owners, and freshness in one place, while a **business glossary** aligns what terms actually mean. Governance also has to answer compliance questions: how PII/PHI gets masked or tokenized, how access is audited, how GDPR deletion requests get honored across raw and derived layers, and how retention policies get enforced with expiry and lifecycle rules.

## Interview Questions

<QuestionList topic="Data Governance" storagePrefix="data-governance">

<QuestionCard id="dg-01" number={1} question="What is data governance and why does it matter?" difficulty="easy" topic="Fundamentals">
Data governance is the set of processes, policies, and responsibilities that ensure data is discoverable, trusted, secure, and used correctly. It matters because data platforms scale to many teams; without governance, definitions diverge, access becomes risky, and compliance issues appear. Good governance enables faster, safer self-service analytics.
</QuestionCard>

<QuestionCard id="dg-02" number={2} question="What is the difference between governance and security?" difficulty="medium" topic="Fundamentals">
Security focuses on preventing unauthorized access and ensuring confidentiality/integrity (access controls, encryption). Governance includes security but also covers ownership, definitions, quality, lifecycle, and change management. Governance answers "who owns this data and what does it mean," not just "who can access it."
</QuestionCard>

<QuestionCard id="dg-03" number={3} question="What are common roles in data governance (owner/steward/custodian)?" difficulty="easy" topic="Roles">
Common roles are:

- **data owner** — accountable for the dataset and its business definition
- **data steward** — maintains quality, metadata, and processes
- **data custodian** — operates the technical platform (storage, access, backups)

Clear roles prevent "nobody owns it" incidents.
</QuestionCard>

<QuestionCard id="dg-04" number={4} question="What is a data catalog and what should it contain?" difficulty="easy" topic="Cataloging">
A data catalog is an inventory of datasets with searchable metadata. It typically contains dataset descriptions, owners, lineage, classifications, freshness/SLA, schema, and links to dashboards. A good catalog reduces time to discovery and improves trust and reuse.
</QuestionCard>

<QuestionCard id="dg-05" number={5} question="What is business glossary vs. technical metadata?" difficulty="medium" topic="Metadata">
A business glossary defines business terms (for example "active user," "net revenue") and their approved definitions. Technical metadata describes datasets and fields (schemas, types, partitions, lineage). Both are needed: glossary aligns meaning, technical metadata enables implementation and discovery.
</QuestionCard>

<QuestionCard id="dg-06" number={6} question="RBAC vs. ABAC: what is the difference?" difficulty="medium" topic="Access Control">
RBAC (Role-Based Access Control) grants permissions to roles (analyst, engineer) and assigns users to roles. ABAC (Attribute-Based Access Control) uses attributes (department, region, sensitivity) to evaluate policies dynamically. ABAC is more flexible but can be harder to manage without good tooling.
</QuestionCard>

<QuestionCard id="dg-07" number={7} question="What is row-level and column-level security?" difficulty="medium" topic="Access Control">
Row-level security restricts which rows a user can see (for example only their region). Column-level security restricts which columns are visible (for example hide salary or PII). These controls are important for multi-tenant analytics and compliance.
</QuestionCard>

<QuestionCard id="dg-08" number={8} question="How do you handle PII/PHI data in analytics platforms?" difficulty="hard" topic="Compliance">
You classify sensitive fields, minimize exposure, and enforce least privilege. Common controls include encryption, masking/tokenization, column-level access, and auditing. You also ensure downstream marts only include necessary fields and implement secure sharing patterns.
</QuestionCard>

<QuestionCard id="dg-09" number={9} question="What is data masking and tokenization?" difficulty="medium" topic="Compliance">
Masking hides sensitive values (for example showing only the last 4 digits) while keeping the dataset usable. Tokenization replaces sensitive values with tokens that can be detokenized only with access to a secure mapping. Tokenization is useful when analytics needs stable identifiers without exposing raw PII.
</QuestionCard>

<QuestionCard id="dg-10" number={10} question="How do you implement auditability for data access?" difficulty="medium" topic="Auditing">
You log access events (who queried what, when, from where) and retain logs per policy. You also track permission changes and administrative actions. Audit logs should be queryable for investigations and compliance reporting.
</QuestionCard>

<QuestionCard id="dg-11" number={11} question="How do you handle GDPR “right to be forgotten” in a lakehouse/warehouse?" difficulty="hard" topic="Compliance">
You need a process to delete or anonymize personal data across raw, derived, and serving layers. This can involve targeted deletes (where supported), reprocessing from raw logs without the user, or storing personal data separately to allow deletion without rewriting large datasets. You also document what is feasible given retention and legal constraints.
</QuestionCard>

<QuestionCard id="dg-12" number={12} question="What are data retention policies and how do you enforce them?" difficulty="medium" topic="Retention">
Retention policies define how long data is kept and when it must be deleted or archived. Enforcement can be implemented with partition expiry jobs, lifecycle rules on object storage, snapshot expiration, and access reviews. Retention must align with legal requirements and business needs.
</QuestionCard>

</QuestionList>

## Common Interview Scenarios

- **"A new analyst asks to see raw customer PII 'just to debug a report' — how do you respond?"** Tests least privilege, masking/tokenization, and role boundaries.
- **"Legal asks you to delete a specific user's data across the entire warehouse for a GDPR request — walk through your approach."** Tests the raw/derived layer deletion strategy and audit trail.
- **"Two teams have different definitions of 'active user' in their dashboards — how do you fix that at the platform level?"** Tests business glossary thinking and conformed definitions.
- **"Design the ownership model for a new data domain with three producing teams and ten consuming teams."** Tests owner/steward/custodian role design.
